MCP server 为代理暴露结构化工具。请使用它们，而不是让代理从文字中猜 CLI flags，或手动复制长命令。

## 推荐工作流

### 新项目

1. `bfs_get_guidance`
2. `bfs_get_schema`
3. `bfs_check_compatibility`
4. `bfs_plan_project`
5. `bfs_create_project`

### 现有项目

1. `bfs_get_guidance`
2. `bfs_plan_addition`
3. `bfs_add_feature`

<Callout kind="warn" title="变更行为">
`bfs_plan_project` 只做 dry-run。`bfs_create_project` 会把新项目写入磁盘。`bfs_add_feature` 会修改现有项目。通过 MCP 创建项目或添加功能时不会安装依赖。
</Callout>

<Callout kind="info" title="图元数据">
项目规划、创建和功能添加工具都会返回 `graphSummary`、`effectiveStack` 和 `stackPartSpecs`，让代理可以检查由 graph 推导出的技术栈视图，并与 `bts.jsonc` 中持久化的内容保持一致。
</Callout>

## 工具参考

### `bfs_get_guidance`

| 字段 | 详情 |
| --- | --- |
| 用途 | 返回工作流规则、字段语义、歧义处理规则和关键约束。 |
| 何时调用 | 首先调用，在其他 Better Fullstack MCP 工具之前。 |
| 写入文件 | 否。 |
| 必需输入 | 无。 |
| 可选输入 | 无。 |
| 输出 | JSON 指南，覆盖推荐工具顺序、数组字段、生态规则和示例。 |

```json
{}
```

### `bfs_get_schema`

| 字段 | 详情 |
| --- | --- |
| 用途 | 返回某个分类或所有分类的有效选项值。 |
| 何时调用 | 在规划 stack 前调用，尤其当用户以非正式方式命名技术时。 |
| 写入文件 | 否。 |
| 必需 inputs | 无。 |
| 可选输入 | `category`, `ecosystem`. |
| 输出 | 按 category 或 ecosystem 过滤的 JSON schema/options 数据。 |

```json
{
  "category": "database",
  "ecosystem": "typescript"
}
```

### `bfs_check_compatibility`

| 字段 | 详情 |
| --- | --- |
| 用途 | 验证技术栈组合，并返回带 warnings 的调整后选择。 |
| 何时调用 | 在 `bfs_plan_project` 或 `bfs_create_project` 之前。 |
| 写入文件 | 否。 |
| 必需输入 | `ecosystem`. |
| 可选输入 | Frontend、backend、runtime、database、ORM、API、auth、services、UI/CSS、mobile fields、addons、examples、package manager、deployment fields 和 ecosystem-specific fields。 |
| 输出 | JSON 兼容性分析，包含 changes、notes 和调整后的技术栈数据。 |

```json
{
  "ecosystem": "typescript",
  "frontend": ["next"],
  "backend": "self",
  "runtime": "none",
  "database": "postgres",
  "orm": "drizzle",
  "api": "trpc"
}
```

### `bfs_plan_project`

| 字段 | 详情 |
| --- | --- |
| 用途 | 在内存中生成项目，并返回文件树摘要。 |
| 何时调用 | 在兼容性验证之后、写入文件之前。 |
| 写入文件 | 否。 |
| 必需输入 | 无，但推荐 `projectName` 和 `ecosystem`。 |
| 可选输入 | 完整项目技术栈选择。 |
| 输出 | JSON 成功标记，加上生成的文件/目录摘要、`graphSummary`、`effectiveStack` 和 `stackPartSpecs`。 |

```json
{
  "projectName": "my-app",
  "ecosystem": "typescript",
  "frontend": ["tanstack-router"],
  "backend": "hono",
  "runtime": "bun",
  "database": "sqlite",
  "orm": "drizzle"
}
```

### `bfs_create_project`

| 字段 | 详情 |
| --- | --- |
| 用途 | 将新生成项目写入磁盘。 |
| 何时调用 | 仅在 `bfs_plan_project` 成功且用户批准后调用。 |
| 写入文件 | 是，会创建项目目录和生成文件。 |
| 必需输入 | `projectName`. |
| 可选输入 | 完整项目技术栈选择。 |
| 输出 | JSON 成功标记、项目目录、文件数量、图元数据、addon warnings 和安装命令指引。 |

```json
{
  "projectName": "my-app",
  "ecosystem": "java",
  "javaWebFramework": "spring-boot",
  "javaBuildTool": "maven",
  "javaTestingLibraries": ["junit5"]
}
```

### `bfs_plan_addition`

| 字段 | 详情 |
| --- | --- |
| 用途 | 读取 `bts.jsonc` 并验证现有项目的拟添加内容。 |
| 何时调用 | 在用 `bfs_add_feature` 修改现有项目前。 |
| 写入文件 | 否。 |
| 必需输入 | `projectDir`. |
| 可选输入 | `addons`, `webDeploy`, `serverDeploy`. |
| 输出 | 现有配置摘要、拟添加内容、二者的图元数据、已存在的 addons 和兼容性 warnings。 |

```json
{
  "projectDir": "/absolute/path/to/my-app",
  "addons": ["mcp", "skills"]
}
```

### `bfs_add_feature`

| 字段 | 详情 |
| --- | --- |
| 用途 | 向现有 Better Fullstack 项目添加受支持的 addons 或部署设置。 |
| 何时调用 | 在 `bfs_plan_addition` 成功且用户批准后。 |
| 写入文件 | 是，会修改现有项目。 |
| 必需输入 | `projectDir`. |
| 可选输入 | `addons`, `webDeploy`, `serverDeploy`, `packageManager`. |
| 输出 | add handler 的 JSON 结果、更新后的图元数据，以及可用时的安装指引。 |

```json
{
  "projectDir": "/absolute/path/to/my-app",
  "addons": ["biome"],
  "packageManager": "bun"
}
```

## 代理规则

- 将 `frontend`、`addons`、`examples`、`rustLibraries`、`pythonAi`、`javaLibraries` 和 `javaTestingLibraries` 视为数组。
- 在选择生态专属字段前设置 `ecosystem`。
- 当用户明确希望禁用某个分类时，优先使用 `none`。
- 通过 MCP 创建项目目前不会暴露每个 CLI workflow flag。例如，创建项目期间 `aiDocs` 使用 server 内置默认值。
- 在用户批准 dry-run 或添加计划之前，不要调用写入工具。
- 在 MCP 创建或添加功能后告诉用户要运行哪些安装/测试命令，因为 MCP 不会安装依赖。
