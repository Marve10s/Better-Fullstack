The Better Fullstack MCP server lets AI agents scaffold projects through structured tool calls instead of guessing CLI flags. It handles schema lookup, compatibility validation, dry-run previews, and project creation.

Works with Claude Code, Cursor, VS Code Copilot, Windsurf, Claude Desktop, Zed, Cline, and any MCP-compatible client.

## Setup

### Claude Code

```bash
claude mcp add --transport stdio better-fullstack -- npx create-better-fullstack mcp
```

### Cursor

Create or edit `.cursor/mcp.json` in your project (or globally at `~/.cursor/mcp.json`):

```json
{
  "mcpServers": {
    "better-fullstack": {
      "command": "npx",
      "args": ["-y", "create-better-fullstack", "mcp"]
    }
  }
}
```

### VS Code

## Tools

### Discovery

| Tool | Purpose |
| --- | --- |
| `bfs_get_guidance` | Returns workflow rules, field semantics, and critical constraints. **Call this first.** |
| `bfs_get_schema` | Returns valid options for any or all configuration categories (50+ categories across all ecosystems). |
| `bfs_check_compatibility` | Validates a stack combination and returns auto-adjusted selections with warnings. |

### Project creation

| Tool | Purpose |
| --- | --- |
| `bfs_plan_project` | Dry-run — generates a project in memory and returns the file tree without writing to disk. |
| `bfs_create_project` | Scaffolds a new project to disk. Requires `projectName` (kebab-case). |

### Existing projects

| Tool | Purpose |
| --- | --- |
| `bfs_plan_addition` | Validates proposed addons against an existing project's `bts.jsonc` config. |
| `bfs_add_feature` | Adds addons or features to an existing project. |

<Callout kind="info" title="Dependencies are never installed in MCP mode">
The server skips `npm install` / `bun install` to avoid timeout errors. After scaffolding, tell the user to run the install command manually: `cd <project> && bun install` (or the package manager they chose).
</Callout>

## Resources

The server exposes three MCP resources that clients can read directly:

| Resource | Contents |
| --- | --- |
| `docs://compatibility-rules` | Which frontend/backend/API/ORM combinations are valid or auto-adjusted. |
| `docs://stack-options` | All available technology options per category across every ecosystem. |
| `docs://getting-started` | Quick-start recipes for TypeScript, Rust, Python, Go, and Java. |

## Recommended tool order

For a new project:

1. `bfs_get_guidance` — understand rules and field semantics
2. `bfs_get_schema` — fetch valid options for your planned categories
3. `bfs_check_compatibility` — validate and auto-fix your stack
4. `bfs_plan_project` — dry-run to confirm the file tree
5. `bfs_create_project` — write to disk

For an existing project:

1. `bfs_plan_addition` — validate proposed changes
2. `bfs_add_feature` — apply them

## Key rules for agents

- **`frontend` is an array** — supports multiple frontends in one monorepo, e.g., `["next", "native-bare"]`.
- **`"none"` means skip** — it disables a feature entirely, not "use the default."
- **Set `ecosystem` first** — it determines which fields apply. TypeScript fields are ignored for Rust/Python/Go/Java and vice versa.
- **Array fields** — `frontend`, `addons`, `examples`, `aiDocs`, `rustLibraries`, `pythonAi`, `javaLibraries`, `javaTestingLibraries` all accept multiple values.
- **Always run `bfs_check_compatibility` before creating** — it auto-corrects invalid combinations (e.g., tRPC + Svelte → oRPC) and returns specific warnings.

Paste this into your agent's system prompt or context to guide it:

```text
Use Better Fullstack MCP. First call bfs_get_guidance to understand
```
