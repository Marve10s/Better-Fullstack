---
title: Referencia de herramientas MCP
description: Referencia detallada de herramientas MCP de Better Fullstack, inputs, outputs y flujos seguros.
---

El servidor MCP expone herramientas estructuradas para agentes. Úsalas en lugar de pedirle a un agente que adivine flags de CLI desde prosa o copie comandos largos a mano.

## Flujos recomendados

### Proyecto nuevo

1. `bfs_get_guidance`
2. `bfs_get_schema`
3. `bfs_check_compatibility`
4. `bfs_plan_project`
5. `bfs_create_project`

### Proyecto existente

1. `bfs_get_guidance`
2. `bfs_plan_addition`
3. `bfs_add_feature`

<Callout kind="warn" title="Comportamiento de mutación">
`bfs_plan_project` es solo dry-run. `bfs_create_project` escribe un proyecto nuevo en disco. `bfs_add_feature` muta un proyecto existente. La creación de proyectos y la adición de funcionalidades vía MCP no instalan dependencias.
</Callout>

<Callout kind="info" title="Metadata de grafo">
Las herramientas de planificación, creación y adición de proyecto incluyen `graphSummary`, `effectiveStack` y `stackPartSpecs` para que los agentes inspeccionen la misma vista de stack derivada del grafo que persiste `bts.jsonc`.
</Callout>

## Referencia de herramientas

### `bfs_get_guidance`

| Campo | Detalles |
| --- | --- |
| Propósito | Devuelve reglas de flujo, semántica de campos, reglas de ambigüedad y restricciones críticas. |
| Cuándo llamarla | Primero, antes de otras herramientas MCP de Better Fullstack. |
| Escribe archivos | No. |
| Inputs requeridos | Ninguno. |
| Inputs opcionales | Ninguno. |
| Output | Guía JSON que cubre el orden recomendado de herramientas, campos array, reglas de ecosistema y ejemplos. |

```json
{}
```

### `bfs_get_schema`

| Campo | Detalles |
| --- | --- |
| Propósito | Devuelve valores de opción válidos para una categoría o todas las categorías. |
| Cuándo llamarla | Antes de planificar un stack, especialmente cuando el usuario nombró tecnologías informalmente. |
| Escribe archivos | No. |
| Inputs requeridos | Ninguno. |
| Inputs opcionales | `category`, `ecosystem`. |
| Output | Datos JSON de esquema/opciones filtrados por categoría o ecosistema. |

```json
{
  "category": "database",
  "ecosystem": "typescript"
}
```

### `bfs_check_compatibility`

| Campo | Detalles |
| --- | --- |
| Propósito | Valida una combinación de stack y devuelve selecciones ajustadas con warnings. |
| Cuándo llamarla | Antes de `bfs_plan_project` o `bfs_create_project`. |
| Escribe archivos | No. |
| Inputs requeridos | `ecosystem`. |
| Inputs opcionales | Frontend, backend, runtime, database, ORM, API, auth, servicios, UI/CSS, campos mobile, addons, examples, package manager, campos de deployment y campos específicos de ecosistema. |
| Output | Análisis JSON de compatibilidad con cambios, notas y datos de stack ajustados. |

```json
{
  "ecosystem": "typescript",
  "frontend": ["next"],
  "backend": "self",
  "runtime": "none",
  "database": "postgres",
  "orm": "drizzle",
  "api": "trpc"
}
```

### `bfs_plan_project`

| Campo | Detalles |
| --- | --- |
| Propósito | Genera un proyecto en memoria y devuelve un resumen del árbol de archivos. |
| Cuándo llamarla | Después de validar compatibilidad y antes de escribir archivos. |
| Escribe archivos | No. |
| Inputs requeridos | Ninguno, pero se recomiendan `projectName` y `ecosystem`. |
| Inputs opcionales | Selecciones completas del stack de proyecto. |
| Output | Indicador JSON de éxito más resumen de archivos/directorios generados, `graphSummary`, `effectiveStack` y `stackPartSpecs`. |

```json
{
  "projectName": "my-app",
  "ecosystem": "typescript",
  "frontend": ["tanstack-router"],
  "backend": "hono",
  "runtime": "bun",
  "database": "sqlite",
  "orm": "drizzle"
}
```

### `bfs_create_project`

| Campo | Detalles |
| --- | --- |
| Propósito | Escribe un proyecto generado nuevo en disco. |
| Cuándo llamarla | Solo después de que `bfs_plan_project` tenga éxito y el usuario apruebe. |
| Escribe archivos | Sí, crea el directorio del proyecto y los archivos generados. |
| Inputs requeridos | `projectName`. |
| Inputs opcionales | Selecciones completas del stack del proyecto. |
| Output | Indicador JSON de éxito, directorio del proyecto, conteo de archivos, metadatos de grafo, warnings de addons y guía del comando de instalación. |

```json
{
  "projectName": "my-app",
  "ecosystem": "java",
  "javaWebFramework": "spring-boot",
  "javaBuildTool": "maven",
  "javaTestingLibraries": ["junit5"]
}
```

### `bfs_plan_addition`

| Campo | Detalles |
| --- | --- |
| Propósito | Lee `bts.jsonc` y valida adiciones propuestas para un proyecto existente. |
| Cuándo llamarla | Antes de mutar un proyecto existente con `bfs_add_feature`. |
| Escribe archivos | No. |
| Inputs requeridos | `projectDir`. |
| Inputs opcionales | `addons`, `webDeploy`, `serverDeploy`. |
| Output | Resumen de configuración existente, adiciones propuestas, metadatos de grafo para ambas, addons ya presentes y warnings de compatibilidad. |

```json
{
  "projectDir": "/absolute/path/to/my-app",
  "addons": ["mcp", "skills"]
}
```

### `bfs_add_feature`

| Campo | Detalles |
| --- | --- |
| Propósito | Agrega addons soportados o ajustes de deployment a un proyecto Better Fullstack existente. |
| Cuándo llamarla | Después de que `bfs_plan_addition` tenga éxito y el usuario apruebe. |
| Escribe archivos | Sí, muta el proyecto existente. |
| Inputs requeridos | `projectDir`. |
| Inputs opcionales | `addons`, `webDeploy`, `serverDeploy`, `packageManager`. |
| Output | Resultado JSON del handler add, metadatos de grafo después de la actualización y guía de instalación cuando esté disponible. |

```json
{
  "projectDir": "/absolute/path/to/my-app",
  "addons": ["biome"],
  "packageManager": "bun"
}
```

## Reglas para agentes

- Trata `frontend`, `addons`, `examples`, `rustLibraries`, `pythonAi`, `javaLibraries` y `javaTestingLibraries` como arrays.
- Configura `ecosystem` antes de seleccionar campos específicos de ecosistema.
- Prefiere `none` cuando el usuario quiera deshabilitar explícitamente una categoría.
- La creación de proyectos MCP actualmente no expone todos los flags de flujo de CLI. Por ejemplo, `aiDocs` usa el default incorporado del servidor durante la creación del proyecto.
- No llames herramientas de escritura hasta que el usuario apruebe el dry-run o el plan de adición.
- Di al usuario qué comandos de instalación/test ejecutar después de la creación o adición MCP, porque MCP no instala dependencias.
