title: Referencia de herramientas MCP
description: Referencia detallada de herramientas MCP de Better Fullstack, inputs, outputs y flujos seguros.
---

El servidor MCP expone herramientas estructuradas para agentes. Úsalas en lugar de pedirle a un agente que adivine flags de CLI desde prosa o copie comandos largos a mano.

## Flujos recomendados

### Proyecto nuevo

1. `bfs_get_guidance`
2. `bfs_get_schema`
3. `bfs_check_compatibility`
4. `bfs_plan_project`
5. `bfs_create_project`

### Proyecto existente

1. `bfs_get_guidance`
2. `bfs_plan_addition`
3. `bfs_add_feature`

<Callout kind="warn" title="Comportamiento de mutación">
`bfs_plan_project` es solo dry-run. `bfs_create_project` escribe un proyecto nuevo en disco. `bfs_add_feature` muta un proyecto existente. La creación de proyecto y adición de features vía MCP no instalan dependencias.
</Callout>

<Callout kind="info" title="Metadata de grafo">
Las herramientas de planificación, creación y adición de proyecto incluyen `graphSummary`, `effectiveStack` y `stackPartSpecs` para que los agentes inspeccionen la misma vista de stack derivada del grafo que persiste `bts.jsonc`.
</Callout>

## Referencia de herramientas

### `bfs_get_guidance`

| Campo | Detalles |
| --- | --- |
| Proposito | Devuelve reglas de workflow, semantica de campos, reglas de ambiguedad y restricciones criticas. |
| Cuando llamarla | Primero, antes de otras herramientas MCP de Better Fullstack. |
| Escribe archivos | No. |
| Inputs requeridos | Ninguno. |
| Inputs opcionales | Ninguno. |
| Output | Guia JSON que cubre orden recomendado de herramientas, campos array, reglas de ecosistema y ejemplos. |

```json
{}
```

### `bfs_get_schema`

| Campo | Detalles |
| --- | --- |
| Proposito | Devuelve valores de opcion validos para una categoria o todas las categorias. |
| Cuando llamarla | Antes de planificar un stack, especialmente cuando el usuario nombro tecnologias informalmente. |
| Escribe archivos | No. |
| Inputs requeridos | Ninguno. |
| Inputs opcionales | `category`, `ecosystem`. |
| Output | Datos JSON de schema/opciones filtrados por categoria o ecosistema. |

```json
{
  "category": "database",
  "ecosystem": "typescript"
}
```

### `bfs_check_compatibility`

| Campo | Detalles |
| --- | --- |
| Proposito | Valida una combinacion de stack y devuelve selecciones ajustadas con warnings. |
| Cuando llamarla | Antes de `bfs_plan_project` o `bfs_create_project`. |
| Escribe archivos | No. |
| Inputs requeridos | `ecosystem`. |
| Inputs opcionales | Frontend, backend, runtime, database, ORM, API, auth, servicios, UI/CSS, campos mobile, addons, examples, package manager, campos de deployment y campos especificos de ecosistema. |
| Output | Analisis JSON de compatibilidad con cambios, notas y datos de stack ajustados. |

```json
{
  "ecosystem": "typescript",
  "frontend": ["next"],
  "backend": "self",
  "runtime": "none",
  "database": "postgres",
  "orm": "drizzle",
  "api": "trpc"
}
```

### `bfs_plan_project`

| Campo | Detalles |
| --- | --- |
| Proposito | Genera un proyecto en memoria y devuelve un resumen del arbol de archivos. |
| Cuando llamarla | Despues de validar compatibilidad y antes de escribir archivos. |
| Escribe archivos | No. |
| Inputs requeridos | Ninguno, pero se recomiendan `projectName` y `ecosystem`. |
| Inputs opcionales | Selecciones completas del stack de proyecto. |
| Output | Flag JSON de exito mas resumen de archivos/directorios generados, `graphSummary`, `effectiveStack` y `stackPartSpecs`. |

```json
{
  "projectName": "my-app",
  "ecosystem": "typescript",
  "frontend": ["tanstack-router"],
  "backend": "hono",
  "runtime": "bun",
  "database": "sqlite",
  "orm": "drizzle"
}
```

### `bfs_create_project`

| Campo | Detalles |
| --- | --- |
| Proposito | Escribe un proyecto generado nuevo en disco. |
| Cuando llamarla | Solo despues de que `bfs_plan_project` tenga exito y el usuario apruebe. |
| Escribe archivos | Si, crea el directorio del proyecto y los archivos generados. |
| Inputs requeridos | `projectName`. |
| Inputs opcionales | Selecciones completas del stack de proyecto. |
| Output | Flag JSON de exito, directorio del proyecto, conteo de archivos, metadata de grafo, warnings de addons y guia de comando de install. |

```json
{
  "projectName": "my-app",
  "ecosystem": "java",
  "javaWebFramework": "spring-boot",
  "javaBuildTool": "maven",
  "javaTestingLibraries": ["junit5"]
}
```

### `bfs_plan_addition`

| Campo | Detalles |
| --- | --- |
| Proposito | Lee `bts.jsonc` y valida adiciones propuestas para un proyecto existente. |
| Cuando llamarla | Antes de mutar un proyecto existente con `bfs_add_feature`. |
| Escribe archivos | No. |
| Inputs requeridos | `projectDir`. |
| Inputs opcionales | `addons`, `webDeploy`, `serverDeploy`. |
| Output | Resumen de config existente, adiciones propuestas, metadata de grafo para ambas, addons ya presentes y warnings de compatibilidad. |

```json
{
  "projectDir": "/absolute/path/to/my-app",
  "addons": ["mcp", "skills"]
}
```

### `bfs_add_feature`

| Campo | Detalles |
| --- | --- |
| Proposito | Agrega addons soportados o ajustes de deployment a un proyecto Better Fullstack existente. |
| Cuando llamarla | Despues de que `bfs_plan_addition` tenga exito y el usuario apruebe. |
| Escribe archivos | Si, muta el proyecto existente. |
| Inputs requeridos | `projectDir`. |
| Inputs opcionales | `addons`, `webDeploy`, `serverDeploy`, `packageManager`. |
| Output | Resultado JSON del handler add, metadata de grafo despues de la actualizacion y guia de install cuando este disponible. |

```json
{
  "projectDir": "/absolute/path/to/my-app",
  "addons": ["biome"],
  "packageManager": "bun"
}
```

## Reglas para agentes

- Trata `frontend`, `addons`, `examples`, `rustLibraries`, `pythonAi`, `javaLibraries` y `javaTestingLibraries` como arrays.
- Configura `ecosystem` antes de seleccionar campos especificos de ecosistema.
- Prefiere `none` cuando el usuario quiera deshabilitar explicitamente una categoria.
- La creacion de proyectos MCP actualmente no expone todos los flags de workflow de CLI. Por ejemplo, `aiDocs` usa el default incorporado del servidor durante la creacion de proyecto.
- No llames herramientas de escritura hasta que el usuario apruebe el dry-run o el plan de adicion.
- Di al usuario que comandos de install/test ejecutar despues de creacion o adicion MCP, porque MCP no instala dependencias.
