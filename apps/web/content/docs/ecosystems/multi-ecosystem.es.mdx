---
title: Multi-Ecosystem
description: Compone un proyecto Better Fullstack desde partes frontend, backend, base de datos y mobile entre ecosistemas.
---

El modo Multi-Ecosystem modela un proyecto como partes de stack conectadas en lugar de una seleccion plana de ecosistema. Úsalo cuando un producto cruza lenguajes, por ejemplo un frontend TypeScript, una API Go, una base de datos PostgreSQL compartida y una app mobile React Native.

Abrelo en el [Stack Builder](https://better-fullstack.dev/new?mode=multi) o pasa flags `--part` repetidos a la CLI.

## Partes del stack

Cada parte usa esta forma:

```txt
role:ecosystem:tool
owner.role:ecosystem:tool
```

Los roles de primer nivel describen apps generadas o infraestructura compartida. Los roles con propietario adjuntan una capacidad a una parte padre.

| Ejemplo | Significado |
| --- | --- |
| `frontend:typescript:next` | Genera un frontend TypeScript con Next.js. |
| `backend:go:gin` | Genera un backend Go Gin. |
| `backend.orm:go:gorm` | Adjunta GORM al backend seleccionado. |
| `backend.api:rust:tonic` | Adjunta Tonic gRPC a un backend Rust. |
| `database:universal:postgres` | Agrega una parte de base de datos PostgreSQL compartida. |
| `mobile:react-native:native-bare` | Genera una app mobile React Native. |

## Ejemplo con script

```npm
npm create better-fullstack@latest my-mixed-app -- \
  --part frontend:typescript:next \
  --part backend:go:gin \
  --part backend.orm:go:gorm \
  --part backend.auth:go:jwt \
  --part database:universal:postgres \
  --part mobile:react-native:native-bare \
  --addons turborepo docker-compose
```

## Fuente de verdad

Los proyectos multi-ecosystem escriben `stackParts` en `bts.jsonc`. Esas partes son la fuente de verdad para la estructura generada, el historial y los comandos reproducibles. Campos legacy como `backend`, `orm` o `frontend` aun pueden aparecer como fallbacks de compatibilidad derivados del grafo.

Los comandos generados preservan el grafo como flags `--part` repetidos, para que el proyecto pueda recrearse sin depender de estado oculto del builder.

## Flujo del builder

El modo Multi-Ecosystem del Stack Builder recorre:

1. Frontend - frontends web TypeScript.
2. Backend - backends TypeScript, Rust, Python, Go, Java, Elixir o .NET y capacidades de backend con propietario.
3. Database - opciones universales de base de datos compartida.
4. Mobile - opciones de app React Native.
5. Finalize - addons, docs de IA, install, Git y ajustes de version.

## Compatibilidad

- El modelo de grafo actual soporta solo una parte primaria seleccionada por rol.
- Las partes de capacidad con propietario deben referenciar un owner, como `backend.orm:go:gorm`.
- Los ecosistemas de capacidades normalmente deben coincidir con el ecosistema de su owner.
- Las partes universales de base de datos se pueden compartir entre ecosistemas.
- Algunas herramientas aportan partes implicitas. Por ejemplo, los backends seleccionados pueden aportar supuestos de base de datos o routing que aparecen en los `stackParts` generados.
- El compatibility checker valida las partes del grafo antes de generar y reporta problemas de owner, duplicados y herramientas no soportadas.
