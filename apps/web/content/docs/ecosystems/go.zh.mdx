Go 项目使用 Go modules 和 Go 原生模板。为 HTTP 服务、gRPC、CLI、日志、ORM 和 Go 认证辅助代码选择 Go 选项。

## 前置要求

- Go 工具链。
- 如果要初始化仓库，需要 Git。

## 脚本化示例

```npm
npm create better-fullstack@latest my-go-api -- \
  --ecosystem go \
  --go-web-framework gin \
  --go-orm gorm \
  --go-logging zap \
  --go-auth jwt \
  --go-api none \
  --go-cli none \
  --go-testing testify \
  --go-realtime gorilla-websocket \
  --go-message-queue nats \
  --go-caching redis \
  --go-config viper \
  --go-observability opentelemetry \
  --auth none \
  --ai-docs none \
  --version-channel stable \
  --no-install \
  --no-git
```

## Go 分类

| 分类 | 值 |
| --- | --- |
| Web 框架 | `gin` `echo` `fiber` `chi` `none` |
| ORM | `gorm` `sqlc` `ent` `none` |
| API | `grpc-go` `gqlgen` `none` |
| CLI | `cobra` `bubbletea` `urfave-cli` `none` |
| 日志 | `zap` `zerolog` `slog` `logrus` `none` |
| Go 认证辅助代码 | `casbin` `jwt` `goth` `none` |
| 测试（多选） | `testify` `gomock` `none` |
| 实时能力 | `gorilla-websocket` `centrifuge` `none` |
| 消息队列 | `nats` `watermill` `none` |
| 缓存 | `redis` `ristretto` `none` |
| 配置 | `viper` `koanf` `none` |
| 可观测性 | `opentelemetry` `none` |

## Auth 拆分

Go 有两个与认证相关的字段：

- `--go-auth casbin|jwt|none` 添加 Go 原生认证辅助代码。
- `--auth go-better-auth` 通过全局 auth 字段选择 GoBetterAuth。

Clerk、NextAuth、Stack Auth、Supabase Auth 和 Auth0 等其他全局 auth 提供商偏向 TypeScript，不应作为通用 Go 脚手架支持来展示。

## 生成行为

- 启用安装时，`go mod tidy` 是安装/验证路径。
- gRPC、Cobra 和 Bubble Tea 文件仅在被选中时包含。
- 测试、实时能力、消息队列、缓存、配置和可观测性选择会在 `internal/` 下生成独立包（并为 `testify`/`gomock` 生成示例测试）。
- `gqlgen` 会搭建 `graph/schema.graphqls` + `gqlgen.yml`；运行 `go generate ./graph` 生成 GraphQL 服务器代码（与 Ent 的代码生成流程一致）。
- `goth` 会从 `GOOGLE_CLIENT_ID`/`GITHUB_CLIENT_ID` 形式的环境凭据注册 OAuth 提供商。
- AI 指令文件仍可通过 `--ai-docs` 使用，包括 `--ai-docs none`。
