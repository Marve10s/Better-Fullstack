Las opciones de backend deciden donde corre la lógica de aplicación y como los clientes se comunican con ella.

## Framework de backend

Las opciones de backend TypeScript incluyen Hono, Express, Fastify, Elysia, NestJS, AdonisJS, Nitro, Encore, Convex, feTS, `self` y `none`.

Los ecosistemas de lenguaje exponen sus propias superficies backend:

| Ecosistema | Opciones de backend |
| --- | --- |
| Rust | Axum, Actix Web, Rocket o none. |
| Python | FastAPI, Django, Flask, Litestar o none. |
| Go | Gin, Echo, Fiber, Chi o none. |
| Java | Spring Boot, Quarkus o none. |
| Elixir | Phoenix, Phoenix LiveView o none. |

## Runtime

Los servidores TypeScript pueden apuntar a Bun, Node.js, Workers o ningun runtime separado. Los ecosistemas no TypeScript usan sus toolchains nativas: Cargo, uv, Go modules, Maven/Gradle o Mix.

## Capa de API

Las opciones de API incluyen alternativas TypeScript RPC/GraphQL, Rust Tonic o async-graphql, opciones Django API/GraphQL de Python, Go gRPC y Elixir REST o Absinthe.

## Ownership multi-ecosystem

En modo multi, las capacidades de API, ORM y auth se adjuntan al backend seleccionado. Por ejemplo, `backend.api:rust:tonic` pertenece a un backend Rust, mientras que `backend.orm:go:gorm` pertenece a un backend Go.
