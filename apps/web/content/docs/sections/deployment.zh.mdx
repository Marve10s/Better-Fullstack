---
title: 部署
description: 将 Web 和 server 部署目标作为一个 stack 分区来选择。
---

部署是一个 stack 分区，不是独立的文档支柱。它控制 Better Fullstack 是否在项目创建时通过 `--web-deploy` 和 `--server-deploy` 输出平台文件。

当你希望生成的平台文件与选中的前端、后端、runtime 和 workspace layout 匹配时，请在 scaffold 阶段使用部署 flags。

## 部署 flags

| Flag | 值 |
| --- | --- |
| `--web-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `netlify` `docker` `sst` `none` |
| `--server-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `docker` `sst` `none` |

部署选择会根据 stack 做兼容性检查。一些 fullstack 框架通过 Web app 部署，一些后端不需要单独的 server deployment，而 Convex 或 Encore 会管理自己的平台路径。

## 目标

| 目标 | 最适合 | 说明 |
| --- | --- | --- |
| Docker | Self-hosting、可复现的生产镜像和本地服务编排。 | 与 `docker-compose` addon 分开，后者专注于本地服务。 |
| Vercel | 受支持的 React、Nuxt、SvelteKit、Solid 和 serverless 部署。 | Fullstack 前端通常通过 Web app 部署，而不是单独的 server target。 |
| Cloudflare | Workers runtime stack 和 edge-first Hono servers。 | 兼容性限定在 Workers-compatible server runtimes。 |
| Fly.io | Docker-backed app hosting，支持区域放置和 long-running services。 | 适合持久服务端进程。 |
| Railway | Git-connected app hosting，基于 Dockerfile 构建并提供 managed services。 | 部署前配置生产环境变量。 |
| Render | Blueprint-driven app hosting，基于 Dockerfile 构建（repo 根目录下的 `render.yaml`）。 | Web 和 server 作为独立 Docker services 部署；在 Render dashboard 设置 env vars。 |
| Netlify | 静态和 SSR Web 前端（Next.js、Vite SPAs、SvelteKit、Nuxt）。 | 仅 Web 目标；将 site base directory 设置为 `apps/web`。 |
| SST | 面向 AWS 的 infrastructure as code，用于 TypeScript app workflows。 | 当你想显式拥有 AWS 资源时最合适。 |

## 部署前

- 为 auth secrets、database URLs、provider keys 和 public app URLs 设置生产环境变量。
- 在路由流量前，对生产数据库运行迁移。
- 确认生成的平台配置与应接收请求的 app 匹配。
- 将仅本地服务容器、测试凭据和生成日志排除在源码控制之外。

## Scaffold 后添加

`add` 命令可以记录部署 flags，但它目前会先处理 addons。如果没有选择新的 addon，仅部署的 `add` 命令会提前退出。在 deployment-only add flows 扩展之前，优先在 `create` 时选择部署目标。
