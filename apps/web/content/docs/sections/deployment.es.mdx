---
title: Despliegue
description: Elige destinos de despliegue web y servidor como una sola sección de stack.
---

Deployment es una sección de stack, no un pilar separado de docs. Controla si Better Fullstack emite archivos de plataforma durante la creación del proyecto con `--web-deploy` y `--server-deploy`.

Usa flags de deployment al crear el scaffold cuando quieras que los archivos de plataforma generados coincidan con el frontend, backend, runtime y layout de workspace seleccionados.

## Flags de deployment

| Flag | Valores |
| --- | --- |
| `--web-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `netlify` `docker` `sst` `none` |
| `--server-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `docker` `sst` `none` |

Las opciones de deployment se verifican por compatibilidad contra el stack. Algunos frameworks fullstack despliegan mediante la app web, algunos backends no necesitan un deployment de servidor separado, y Convex o Encore gestionan su propia ruta de plataforma.

## Destinos

| Destino | Mejor encaje | Notas |
| --- | --- | --- |
| Docker | Self-hosting, imagenes de produccion reproducibles y orquestacion local de servicios. | Separado del addon `docker-compose`, que se enfoca en servicios locales. |
| Vercel | Deployments soportados de React, Nuxt, SvelteKit, Solid y serverless. | Los frontends fullstack normalmente se despliegan mediante la app web en lugar de un destino de servidor separado. |
| Cloudflare | Stacks con runtime Workers y servidores Hono edge-first. | La compatibilidad se acota a runtimes de servidor compatibles con Workers. |
| Fly.io | Hosting de apps respaldado por Docker con colocacion regional y servicios long-running. | Bueno para procesos de servidor persistentes. |
| Railway | Hosting de apps conectado a Git con builds basados en Dockerfile y servicios gestionados. | Configura variables de entorno de produccion antes de desplegar. |
| Render | Hosting de apps guiado por Blueprint con builds basados en Dockerfile (`render.yaml` en la raiz del repo). | Web y servidor se despliegan como servicios Docker separados; configura env vars en el dashboard de Render. |
| Netlify | Frontends web estaticos y SSR (Next.js, Vite SPAs, SvelteKit, Nuxt). | Destino solo web; configura el directorio base del sitio como `apps/web`. |
| SST | Infrastructure as code orientada a AWS para workflows de apps TypeScript. | Mejor cuando quieres poseer recursos AWS explicitamente. |

## Antes de desplegar

- Configura variables de entorno de produccion para secretos de auth, URLs de base de datos, keys de providers y URLs publicas de app.
- Ejecuta migraciones de base de datos contra la base de produccion antes de enrutar trafico.
- Confirma que la configuración de plataforma generada coincide con la app que debe recibir requests.
- Mantiene contenedores de servicios locales, credenciales de prueba y logs generados fuera del control de versiones.

## Agregar después del scaffold

El comando `add` puede registrar flags de deployment, pero actualmente procesa addons primero. Si no se selecciona ningun addon nuevo, los comandos `add` solo de deployment salen temprano. Prefiere elegir destinos de deployment durante `create` hasta que se amplien los flujos add solo de deployment.
