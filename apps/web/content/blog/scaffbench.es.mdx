Los agentes de código son muy buenos escribiendo código y sorprendentemente malos empezando proyectos. Pide a uno que cree un monorepo fullstack de producción y puede pasar diez minutos escribiendo manifests, lockfiles y config a mano; muchas veces el resultado ni siquiera compila.

ScaffBench mide exactamente eso: un agente en un workspace vacío, una especificación de proyecto y una pregunta dura al final: **¿el proyecto generado se instala y compila?** Ejecutamos las mismas specs por tres caminos de creación para aislar cuánto ayuda el tooling:

- **MCP**: el agente usa el servidor MCP de Better-Fullstack para planificar y crear el proyecto.
- **BF mention**: sin MCP; el prompt apunta al agente a la CLI y docs de Better-Fullstack, y el agente compone el comando `create`.
- **Prompt**: sin Better-Fullstack; el agente escribe todos los archivos desde cero.

Corrimos la suite completa dos veces: **OpenAI Codex CLI** con tres modelos GPT el 10 de junio y **Claude Code** con tres modelos Claude el 12 de junio. Luego agregamos una [barrida ligera](#barrida-ligera-gemini-kilo-y-opencode) con **Gemini CLI**, **Kilo** y **opencode**. En total: 102 ejecuciones y quince modelos.

## Resultados principales

Los números principales vienen de la barrida de Claude Code: 36 ejecuciones, tres modelos, tres caminos de creación y cuatro specs de proyecto. Misma máquina, mismo harness, mismos prompts salvo las instrucciones del modo de creación. La [barrida GPT](#modelos-gpt-en-codex-cli) muestra la misma estructura y se reporta por separado más abajo.

| Camino de creación | Tiempo medio | Mediana | Tokens de salida medios | Builds que pasan |
| --- | ---: | ---: | ---: | ---: |
| MCP | 113.4s | 85.9s | 5,553 | 12/12 (100%) |
| BF mention | 219.6s | 172.1s | 11,060 | 9/9 (100%)* |
| Prompt | 516.2s | 424.0s | 25,859 | 9/12 (75%) |

\* Tres ejecuciones BF mention se excluyen porque fallaron por un bug de nuestro generador, no por una decisión del agente; ver [la política de scoring](#que-cuenta-como-fallo).

Tres cosas destacaron:

1. **MCP fue cerca de 4.6 veces más rápido que prompt-only** en promedio, con unos 4.7 veces menos tokens de salida.
2. **Todas las ejecuciones MCP pasaron validación**, para todos los modelos. Prompt-only pasó 75%.
3. **El tooling comprime las diferencias entre modelos.** En el camino MCP incluso el modelo más pequeño mantuvo 100% de pass rate.

## Metodología

### Los tres caminos de creación

Cada ejecución empieza en un directorio vacío con el mismo prompt base:

```text
You are running in an empty benchmark workspace:
{run_dir}

Create exactly one project directory named `{project_name}`.
Do not ask questions. Do not start a dev server. Do not write outside the
current working directory.
At the end, report the commands you ran and any errors you hit.

Benchmark target: {spec_title}
Requirements:
{spec_requirements}
```

La diferencia entre caminos es la instrucción de creación añadida al prompt. Prompt-only prohíbe explícitamente MCP, la CLI, el sitio web y archivos del repo de Better-Fullstack:

```text
Creation mode: prompt-only.
Do not use the Better-Fullstack MCP server, Better-Fullstack CLI,
Better-Fullstack website, or files from the Better-Fullstack repository.
Create the project from scratch by writing the files and manifests needed
for a runnable starter.
```

MCP recibe la instrucción opuesta:

```text
Creation mode: Better-Fullstack MCP.
Use the Better-Fullstack MCP tools, starting with bfs_get_guidance. Then use
schema/compatibility/plan as needed and call bfs_create_project to create
the project.
```

BF mention queda en medio: sin MCP tools, pero con permiso para leer la CLI y docs y componer un comando no interactivo.

`heavy-ts` es la prueba de estrés. También es donde ocurrió casi todo lo interesante que falló.

### El harness

Dos barridas, un mismo diseño de harness:

- **Barrida Claude:** Claude Code en modo headless (`claude -p`), capturando el stream JSON de eventos por ejecución. Modelos: `claude-fable-5`, `claude-opus-4-8`, `claude-sonnet-4-6`, con esfuerzo de razonamiento por defecto.
- **Barrida GPT:** OpenAI Codex CLI (`codex exec`), capturando el stream JSON de eventos por ejecución. Modelos: `gpt-5.3-codex-spark` (high effort), `gpt-5.4` (medium), `gpt-5.5` (medium).
- **Aislamiento:** cada run recibe un workspace vacío; las runs MCP reciben una configuración MCP estricta con solo el servidor de Better-Fullstack conectado.
- **Timeout:** 900 segundos por run. Un timeout cuenta como fallo del agente.
- **Métricas:** tiempo de pared, uso de tokens según el propio reporte del agente y llamadas MCP extraídas del stream de eventos. El coste en dólares solo se captura en la barrida Claude; nuestro harness de Codex no reporta coste medido.

Hay una asimetría importante: la barrida GPT corrió dos días antes, contra el generador *antes* de corregir los bugs de template que expuso. Por eso las exclusiones por bug del generador golpean más los resultados GPT, y por eso no conviene comparar las dos barridas cabeza a cabeza en pass rate.

### Qué cuenta como pass

La validación corre después de que el agente termina. TypeScript ejecuta `bun install` y el script de build/check/lint/test del proyecto; Python usa `compileall`; Go descarga módulos y debe compilar. Un proyecto solo cuenta como pass si todas las comprobaciones aplicables salen con código cero.

### Qué cuenta como fallo

<Callout kind="info">
  **Política de scoring:** los fallos causados por bugs de nuestro propio generador de templates se excluyen del denominador de pass-rate. El benchmark mide agentes, no convierte nuestros bugs en puntos negativos para ellos. A cambio, publicamos esos bugs en el análisis en lugar de esconderlos.
</Callout>

En la barrida Claude esto importó tres veces: los tres modelos, en el camino BF mention, generaron `heavy-ts` correctamente con la CLI y chocaron con el mismo bug de nuestro template nativo de Expo. Esas tres ejecuciones se excluyen. Los fallos de prompt-only, en cambio, fueron escritos por el agente y cuentan completos.

En la barrida GPT, que corrió contra el generador previo al fix, importó siete veces: `heavy-ts` por MCP y por CLI falló para los tres modelos por la cadena de bugs Storybook/Expo ya corregida, más un scaffold multi-ecosystem cuyo servicio Go salió sin `go.sum`. Los siete se excluyen por la misma razón. Los fallos propios de los agentes GPT, tres builds rotas escritas a mano en prompt-only, sí cuentan completos.

### Coste

La barrida Claude costó **$60.60** en uso medido de API, menos que el coste real porque los dos runs que llegaron al timeout reportan $0 y el harness de Codex no reporta coste. Lo interesante no es tanto el total como la distribución:

- Run válido más barato: **$0.12**, Sonnet 4.6 creando `light-ts` por CLI en 21 segundos.
- Runs válidos más caros: **$3.50–$4.21**, Fable 5 escribiendo proyectos a mano por prompt-only.
- En Sonnet, prompt-only costó cerca de 5 veces más que MCP ($7.11 vs $1.47) y fue unas 6.6 veces más lento, con un pass rate menor.

El patrón se repite en los tres modelos: cuanto más proyecto tiene que escribir el agente por su cuenta, más pagas por un resultado menos fiable.

Lo que añade la barrida GPT:

- **La estructura no depende del vendor.** Para cada modelo GPT, MCP fue el camino más rápido y prompt-only el más caro en tokens de salida: la misma forma que en Claude, con modelos de otro proveedor y otra CLI de agente.
- **Spark está hecho para velocidad, y se nota.** GPT-5.3 Codex Spark promedió 32 segundos por scaffold MCP, las celdas más rápidas de todo el benchmark, pero en prompt-only esa velocidad costó fiabilidad: escribió proyectos en menos de un minuto y solo compiló la mitad.
- **GPT-5.5 es el outlier en prompt-only.** Fue el único modelo de ambas barridas que pasó todas las specs prompt-only, incluido `heavy-ts`; esa run tomó 542 segundos y 28k tokens de salida, frente a 108 segundos por MCP.
- **Los modelos GPT se apoyan más en las herramientas.** Las runs de Codex hicieron 10–34 llamadas MCP por scaffold, frente a 3–10 en Claude, revisando compatibilidad y replanificando más a menudo para llegar al mismo resultado.

La advertencia importante sigue siendo la misma: las dos barridas usaron CLIs de agente distintas, días distintos y versiones distintas del generador. Comparar caminos dentro de una barrida es el experimento; comparar vendors entre barridas, no.

## Barrida ligera: Gemini, Kilo y opencode

Para la siguiente tanda usamos una versión deliberadamente ligera: **una spec (`light-ts`) × tres caminos × una ejecución por celda**. Nueve modelos en tres CLIs, todos el 12 de junio:

- **Gemini CLI**: Gemini 3.1 Pro, el modelo por defecto de la CLI.
- **opencode** (suscripción Go): Kimi K2.6, GLM-5.1, MiniMax M3, Qwen3.7 Max, DeepSeek-V4 Pro.
- **Kilo CLI** (tier gratuito): Step-3.7 Flash, Laguna m.1, Nex N2-Pro.

Mismos prompts, mismo presupuesto de 900 segundos y misma validación que en las barridas principales. Gemini tuvo una repetición accidental de los tres caminos; la conservamos como muestra gratuita de varianza. En total, 30 runs. Un décimo modelo, Nemotron-3 Ultra en el tier gratuito de Kilo, queda excluido porque el endpoint era demasiado lento para mover un agente de forma útil.

La barrida ligera agrega cuatro señales:

- **Los caminos asistidos fueron 18 de 18.** Todos los modelos pasaron tanto MCP como BF mention. Eso incluye Step-3.7 Flash, un modelo pequeño, rápido y gratuito, creando un monorepo fullstack correcto en 16 segundos y 640 tokens de salida vía BF mention.
- **Prompt-only es donde el campo se divide.** Tres de nueve modelos entregaron proyectos rotos: Step-3.7 Flash fijó `pino-http` a una versión inexistente, MiniMax M3 llamó `useContext` a un procedimiento tRPC y chocó con el método del cliente, y Nex N2-Pro escribió un import que Rollup no pudo resolver.
- **La economía se mantiene en un tercer, cuarto y quinto harness.** Todos los runs asistidos terminaron por debajo de 3k tokens de salida. Todos los prompt-only costaron 8k–36k; los 36k de MiniMax M3 fueron la run individual más cara de ScaffBench hasta ahora, para un proyecto que no compiló.
- **Gemini 3.1 Pro fue excelente en esta tarea.** Tuvo la celda BF mention más rápida de todo el benchmark (26.7 segundos, 863 tokens), un flujo MCP limpio de cuatro llamadas y una de las pocas runs prompt-only que pasaron. Su repetición también recuerda que estas celdas de una sola run son direccionales, no definitivas.

Advertencias de la barrida ligera: una run por celda, y 25 de las runs se ejecutaron en paralelo, así que los tiempos de pared tienen ruido de contención. Toma la estructura y el pass/fail como señal; los segundos exactos son aproximados.

## Análisis cualitativo

### MCP mantiene al agente sobre rieles

Las ejecuciones MCP convergieron en la misma secuencia: `bfs_get_guidance`, compatibility check, plan y `bfs_create_project`. La salida es pequeña porque el trabajo del agente se reduce a elegir configuración; la escritura de archivos queda en el generador de forma determinista. Por eso el pass rate no cae al crecer la spec.

### Prompt-only se ahoga en la spec pesada

En el camino prompt-only, `heavy-ts` venció a casi todos. Algunos modelos llegaron al timeout de 15 minutos; otros terminaron y no compilaron. El único outlier claro fue GPT-5.5, que pasó `heavy-ts` prompt-only con 542 segundos y 28k tokens de salida. Los modelos frontier pueden escribir un starter a mano, pero es la forma más lenta, cara y menos fiable de conseguirlo.

### El benchmark también auditó nuestros templates

Validar cada proyecto generado encontró seis bugs de template en nuestro generador, la mayoría escondidos detrás de un mismo síntoma de Storybook/Expo en `heavy-ts`. Cinco se corrigieron y salieron en `create-better-fullstack` 2.0.2 antes de publicar el post. Ejecutar tu propio producto a través de un benchmark de agentes resultó ser una pasada de QA brutalmente efectiva.

Los seis bugs fueron:

1. Los templates de Storybook comprobaban el frontend con igualdad, pero `frontend` es un array.
2. Un desajuste de rutas dejó el paquete de base de datos sin dependencias en modo graph-part.
3. Faltaba `expo-network` en una variante del template nativo.
4. Los paquetes de framework de Storybook 8 ya no reexportan los tipos `Meta`/`StoryObj`; las importaciones tuvieron que moverse a los paquetes del renderer.
5. `@better-auth/expo` necesitaba cuatro peer dependencies de Expo instaladas explícitamente.
6. `app.json` nativo configura un modo de salida web estático que solo funciona con `expo-router`; este sigue abierto y es el bug detrás de las tres runs BF-mention excluidas.

### Carácter de los modelos

Sonnet 4.6 fue el más rápido y barato en todos los caminos, y con tooling no cedió nada en fiabilidad. Fable 5 fue el más deliberado: más tokens, runs más largas y, en prompt-only, esa meticulosidad tampoco bastó para superar el timeout en `heavy-ts`. Opus 4.8 quedó de forma consistente en medio. El ranking no cambió entre caminos; lo que cambió fue la brecha. El tooling iguala: en MCP, la diferencia entre el mejor y el peor modelo fue de 102 segundos; en prompt-only fueron varios minutos y, dos veces, el techo de tiempo.

## Limitaciones

- **Una ejecución por celda en varias partes.** 102 runs bastan para ver estructura, no para intervalos de confianza finos.
- **Las barridas no son head-to-head.** GPT, Claude y la barrida ligera usan CLIs, días, versiones y modelos distintos.
- **Benchmark de nuestra propia herramienta.** La comparacion entre caminos, tooling vs sin tooling, es el hallazgo; no una afirmacion sobre todos los scaffolders.
- **Builds, no features.** Validar build no demuestra que cada flujo de auth o webhook funcione.
- **Coste sub-reportado en timeouts.** Runs matados a 900s reportan $0, lo que favorece artificialmente a prompt-only.

## Conclusión

La conclusion de ScaffBench es simple: los agentes mejoran mucho cuando no tienen que inventar el scaffold. Con MCP o una CLI clara, los modelos pasan más, gastan menos tokens y llegan antes. Prompt-only sigue siendo impresionante, pero para scaffolding fullstack real es la ruta cara y fragil.
