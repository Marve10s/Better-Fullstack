Coding agents 很擅长写代码，却出人意料地不擅长“开始”项目。让一个 agent scaffold 一个生产级 fullstack monorepo，它可能会花十分钟手写 manifests、lockfiles 和 config，最后生成的项目还经常 build 不起来。

ScaffBench 测量的正是这件事：agent 位于空 workspace 中，拿到一个 project spec，最后回答一个硬问题：**生成的项目能 install 并 build 吗？** 我们用三条创建路径运行同一组 specs，隔离 tooling 的帮助：

- **MCP**：agent 使用 Better-Fullstack MCP server 规划并 scaffold 项目。
- **BF mention**：没有 MCP；prompt 指向 Better-Fullstack CLI 和 docs，agent 自己组合 `create` 命令。
- **Prompt**：完全不使用 Better-Fullstack；agent 从零手写所有文件。

完整 suite 跑了两次：6 月 10 日用 **OpenAI Codex CLI** 和三个 GPT models，6 月 12 日用 **Claude Code** 和三个 Claude models。随后又加入 [轻量 sweep](#轻量-sweepgeminikilo-和-opencode)：**Gemini CLI**、**Kilo** 和 **opencode**。总计 102 runs，十五个 models。

## 核心结果

核心数字来自 Claude Code 的一轮 sweep：36 次运行、三个模型、三条创建路径、四份项目规格。同一台机器、同一套 harness、相同 prompts，只有创建模式的说明不同。[GPT sweep](#codex-cli-上的-gpt-models) 呈现出同样的结构，下面单独报告。

| 创建路径 | 平均时间 | 中位时间 | 平均输出 tokens | 通过 build |
| --- | ---: | ---: | ---: | ---: |
| MCP | 113.4s | 85.9s | 5,553 | 12/12 (100%) |
| BF mention | 219.6s | 172.1s | 11,060 | 9/9 (100%)* |
| Prompt | 516.2s | 424.0s | 25,859 | 9/12 (75%) |

\* 三个 BF-mention runs 因为我们自己的 generator bug 而排除，不计入 agent 失败；见 [评分政策](#什么算失败)。

最突出的三点：

1. **MCP 平均约比 prompt-only 快 4.6 倍**，输出 tokens 约少 4.7 倍。
2. **每个 MCP run 都通过验证**，覆盖所有 models。Prompt-only 的 pass rate 是 75%。
3. **Tooling 压缩了 model 差异。** 在 MCP path 上，最小 model 也保持 100% pass rate。

## 方法

### 三条创建路径

每次 run 都从空目录开始，基础 prompt 相同：

```text
You are running in an empty benchmark workspace:
{run_dir}

Create exactly one project directory named `{project_name}`.
Do not ask questions. Do not start a dev server. Do not write outside the
current working directory.
At the end, report the commands you ran and any errors you hit.

Benchmark target: {spec_title}
Requirements:
{spec_requirements}
```

三条路径的区别在于追加的 creation instruction。Prompt-only 明确禁止 MCP、CLI、website 和 Better-Fullstack repo files：

```text
Creation mode: prompt-only.
Do not use the Better-Fullstack MCP server, Better-Fullstack CLI,
Better-Fullstack website, or files from the Better-Fullstack repository.
Create the project from scratch by writing the files and manifests needed
for a runnable starter.
```

MCP 收到相反的指令：

```text
Creation mode: Better-Fullstack MCP.
Use the Better-Fullstack MCP tools, starting with bfs_get_guidance. Then use
schema/compatibility/plan as needed and call bfs_create_project to create
the project.
```

BF mention 位于中间：没有 MCP tools，但可以读取 CLI/docs，并组合 non-interactive `create` command。

### 四个项目 specs

四个 specs 从轻到重：

| Spec | 要求 |
| --- | --- |
| `light-ts` | React + Vite、Bun 上的 Hono、tRPC、SQLite + Drizzle、Tailwind + DaisyUI、Pino、Vitest、Biome，不含 auth/payments |
| `heavy-ts` | Next.js **和** Expo/React Native app、Bun 上的 Hono、oRPC、PostgreSQL + Drizzle、Better Auth、Stripe、Resend、UploadThing、Effect、TanStack Store/Form/Query、Valibot、Vitest + Playwright、Vercel AI SDK、Socket.IO、Inngest、Framer Motion、OpenTelemetry、PostHog、Umami、Sanity、Upstash Redis、Algolia、S3、shadcn/ui、Turborepo、MSW、Storybook、PWA、Docker compose |
| `python-ai` | FastAPI、SQLModel、PostgreSQL、Pydantic、JWT auth、Celery、Strawberry GraphQL、LangChain、OpenAI SDK、LangGraph、CrewAI、Ruff |
| `multi-ecosystem` | TypeScript Next.js frontend、Python FastAPI backend (SQLModel/Pydantic/LangChain/Celery)、Go Gin service (GORM、gRPC、Cobra、Zap)、shared PostgreSQL |

`heavy-ts` 是压力测试，也是这次 benchmark 里几乎所有有意思的失败发生的地方。

### Harness

两轮主 sweep，使用同一套 harness 设计：

- **Claude sweep：** Claude Code headless mode (`claude -p`)，每次 run 捕获 JSON event stream。模型：`claude-fable-5`、`claude-opus-4-8`、`claude-sonnet-4-6`，默认 reasoning effort。
- **GPT sweep：** OpenAI Codex CLI (`codex exec`)，每次 run 捕获 JSON event stream。模型：`gpt-5.3-codex-spark` (high effort)、`gpt-5.4` (medium)、`gpt-5.5` (medium)。
- **隔离：** 每次 run 都拿到一个全新的空 workspace；MCP runs 使用严格 MCP config，只挂 Better-Fullstack server。
- **Timeout：** 每次 run 900 秒。超时算 agent failure。
- **指标：** wall-clock time、agent 自己 usage report 里的 token usage、event stream 里的 MCP tool calls。美元成本只在 Claude sweep 中记录；Codex harness 不报告 metered cost。

有一个重要的不对称：GPT sweep 早两天运行，当时 generator 还没有修掉这轮 benchmark 暴露出的 template bugs。所以 generator-bug exclusions 对 GPT 结果影响更大，也因此不应该把两轮 sweep 的 pass rate 直接横向比较。

### 什么算通过

验证在 agent 结束后运行。TypeScript 执行 `bun install` 和项目自己的 build/check/lint/test script；Python 用 `compileall`；Go 下载 modules 并必须 compile。只有所有适用检查都返回零，才算 pass。

### 什么算失败

<Callout kind="info">
  **评分政策：** 由我们自己的模板生成器 bug 导致的失败，会从 pass-rate 的分母中排除。这个 benchmark 衡量的是 agents，不会把我们的 bug 算成它们的失败。相应地，我们会在分析中公开这些 generator bugs，而不是把它们藏起来。
</Callout>

在 Claude sweep 中，这正好影响了三次：三个模型都在 BF mention 路径上正确通过 CLI scaffold 了 `heavy-ts`，但撞到了我们 Expo native template 中同一个 generator bug。这三次被排除。Prompt-only 的失败则是 agent 自己写出的结果，完整计入。

GPT sweep 运行在修复前的 generator 上，因此影响了七次：三个模型的 `heavy-ts` 在 MCP 和 CLI 两条路径上都因为后来修复的 Storybook/Expo bug chain 失败，另外还有一次 multi-ecosystem scaffold 的 Go service 没有生成 `go.sum`。这七次同样排除。GPT agents 自己造成的失败，也就是 prompt-only 路径上三次手写 build 失败，则完整计入。

### 成本

Claude sweep 的 metered API usage 是 **$60.60**。真实成本会更高一点，因为两个 timeout runs 报告 $0，而 Codex harness 完全不报告成本。比总额更有意思的是分布：

- 最便宜的 passing run：**$0.12**，Sonnet 4.6 通过 CLI 在 21 秒内 scaffold `light-ts`。
- 最贵的 passing runs：**$3.50–$4.21**，Fable 5 在 prompt-only 路径上手写项目。
- 对 Sonnet 来说，prompt path 的成本约为 MCP path 的 5 倍（$7.11 vs $1.47），速度约慢 6.6 倍，pass rate 还更低。

三个模型里都是同一个模式：agent 需要自己写的项目越多，你花的钱越多，结果还越不可靠。

GPT sweep 补充了几个信号：

- **这个结构不依赖某个 vendor。** 每个 GPT model 里，MCP 都是最快 path，prompt-only 都消耗最多 output tokens；这和 Claude 结果是同一个形状，只是换了模型供应商和 agent CLI。
- **Spark 的速度定位很明显。** GPT-5.3 Codex Spark 的 MCP scaffold 平均 32 秒，是整个 benchmark 里最快的单元格。但在 prompt path 上，这种速度牺牲了可靠性：它在一分钟内手写项目，只有一半 build 通过。
- **GPT-5.5 是 prompt path 的 outlier。** 它是两轮主 sweep 中唯一通过所有 prompt-only specs 的模型，包括 `heavy-ts`；那次 run 花了 542 秒和 28k output tokens，而 MCP 只用了 108 秒。
- **GPT models 更频繁地使用工具。** Codex runs 每个 scaffold 调用 10–34 次 MCP，而 Claude 是 3–10 次；它们会更反复地检查兼容性和重新规划，但最终结果相同。

需要重复这个 caveat：两轮 sweep 使用不同 agent harness、不同日期、不同 generator versions。实验比较的是同一 sweep 内的路径；跨 sweep 排 vendor 名次不是这个 benchmark 的目标。

## 轻量 sweep：Gemini、Kilo 和 opencode

下一批 models 使用一个刻意轻量的版本：**一个 spec (`light-ts`) × 三条 paths × 每个 cell 一次 run**。三个 CLIs 上九个 models，全部在 6 月 12 日运行：

- **Gemini CLI**：Gemini 3.1 Pro，CLI 默认模型。
- **opencode** (Go subscription)：Kimi K2.6、GLM-5.1、MiniMax M3、Qwen3.7 Max、DeepSeek-V4 Pro。
- **Kilo CLI** (free model tier)：Step-3.7 Flash、Laguna m.1、Nex N2-Pro。

轻量 sweep 带来四个信号：

- **Assisted paths 是 18/18。** 每个 model 在 MCP 和 BF mention 上都 pass。这里包括 Step-3.7 Flash，一个小而快的免费模型，它通过 BF mention 在 16 秒、640 output tokens 内 scaffold 出了正确的 fullstack monorepo。
- **Prompt-only 是分化点。** 九个 models 中有三个交付了 broken projects：Step-3.7 Flash 把 `pino-http` 固定到不存在的版本，导致 install 失败；MiniMax M3 把 tRPC procedure 命名为 `useContext`，和 tRPC client 内置方法冲突；Nex N2-Pro 写了无法解析的 import，Rollup 失败。
- **经济性在第三、第四、第五个 harness 上仍成立。** 每个 assisted run 都低于 3k output tokens。每个 prompt-only run 都在 8k–36k；MiniMax M3 的 36k 是目前 ScaffBench 中最贵的单次 run，而且项目没有 build 通过。
- **Gemini 3.1 Pro 很适合这项任务。** 它有整个 benchmark 中最快的 BF mention cell（26.7 秒、863 tokens）、一个 textbook 四调用 MCP flow，也是少数 prompt-only 通过的模型之一。它的 repeat sample 也提醒我们：单 run cell 只能看方向，不是最终统计结论。

轻量 sweep 的 caveats：每个 cell 只有一次 run，而且 25 次 run 并发执行，所以 wall-clock time 有 contention noise。把路径结构和 pass/fail 当作信号，具体秒数只当近似值。

## 定性分析

### MCP 让 agents 保持在轨道上

MCP runs 收敛到相同序列：`bfs_get_guidance`、compatibility check、plan、`bfs_create_project`。输出很小，因为 agent 的任务被压缩成选择配置；文件写入由 generator 确定性完成。因此 spec 变大时 pass rate 不会下降。

### Prompt-only 被 heavy spec 淹没

在 prompt-only path 上，`heavy-ts` 击败了几乎所有人。有些 models 撞到 15 分钟 timeout；有些完成了项目但 build 不起来。唯一清晰的例外是 GPT-5.5，它用 542 秒和 28k output tokens 通过了 `heavy-ts` prompt-only。Frontier models 能手写 starter，但这是最慢、最贵、最脆弱的方式。

### Benchmark 也审计了我们的 templates

验证每个 generated project 暴露了我们 generator 中的六个 template bugs，大多隐藏在 `heavy-ts` 的 Storybook/Expo 症状背后。其中五个在本文发布前已修复并随 `create-better-fullstack` 2.0.2 发布。把自己的产品放进 agent benchmark，结果成了一次非常强力的 QA pass。

六个 bugs 是：

1. Storybook templates 用 equality check 判断 frontend，但 `frontend` 实际是 array。
2. 数据库 package path mismatch 导致 DB package 在 graph-part mode 下缺依赖。
3. Native template 的一个 variant 缺少 `expo-network`。
4. Storybook 8 framework packages 不再 re-export `Meta`/`StoryObj` types，imports 必须移到 renderer packages。
5. `@better-auth/expo` 需要显式安装四个 Expo peer dependencies。
6. Native `app.json` 设置了只适用于 `expo-router` 的 static web output mode；这个仍然 open，也是三次 BF-mention excluded runs 背后的 bug。

### 模型性格

Sonnet 4.6 在每条路径上都是最快、最便宜的；有 tooling 时，它在可靠性上没有任何损失。Fable 5 最谨慎：token counts 最高、runs 最长，而在 prompt path 上，这种谨慎仍然没能让 `heavy-ts` 躲过 timeout。Opus 4.8 稳定地处在中间。模型排序没有因为路径而改变；改变的是差距。Tooling 是 equalizer：MCP 上最好和最差模型相差 102 秒；prompt-only 上差距是数分钟，而且两次撞到时间上限。

## 局限

- **多个部分只有每 cell 一次 run。** 102 runs 足以看出结构，不足以给出精细置信区间。
- **各 sweep 不是 head-to-head。** GPT、Claude 和轻量 sweep 使用不同 CLIs、日期、versions 和 models。
- **我们评测的是自己的工具。** tooling vs no tooling 的 path comparison 是发现，不是对所有 scaffolders 的断言。
- **验证的是 builds，不是完整功能。** build 通过不代表每个 auth flow 或 webhook 都正确。
- **timeout 成本被低报。** 900s 被 kill 的 runs 报 $0，这会让 prompt-only 看起来更便宜。

## 结论

ScaffBench 的结论很直接：当 agents 不必发明 scaffold 时，它们表现好得多。通过 MCP 或清晰 CLI，models pass 更多、消耗 tokens 更少、完成更快。Prompt-only 仍然令人印象深刻，但对真实 fullstack scaffolding 来说，它是昂贵且脆弱的路径。

想试试快速路径？把你的 agent 指向 [Better-Fullstack MCP server](/mcp)，然后让它 scaffold 一个重项目。
