[ScaffBench 2](/blog/scaffbench-2) asked the hardest version of one question — can a coding agent
hand-write a real full-stack project from a prompt, with no scaffolder, and have it build? The answer
was "rarely."

ScaffBench 2.1 puts the newest model, **Claude Sonnet 5**, to that test — head-to-head with **Claude
Opus 4.8** at both low and maximum reasoning effort, on an expanded suite of **13 hard specs across
eight ecosystems**. We also ran **Sonnet 4.6**, **GPT-5.3 Codex Spark**, and — via Google's Antigravity
CLI — **Gemini 3.5 Flash** at high effort, for a six-config field. The prompt path is the purest
measure of raw capability: the agent reads the spec and writes every file itself.

All three configurations have now finished the full 13-spec run. Here's where things stand.

## Claude Sonnet 5: strong instincts, fewer builds

Sonnet 5 selects the right stack as well as Opus — **96% wired, matching it spec-for-spec** — but it
turns fewer of those into projects that compile: **2 of 10, versus Opus's 3 of 11**. And the two aren't
strangers to the same specs: each model uniquely builds one the other can't.

| Spec | Ecosystem | Opus 4.8 · max | Sonnet 5 · max |
| --- | --- | :---: | :---: |
| `python-ingestion-api` | Python | ✅ builds | ✅ builds |
| `rust-leptos-axum` | Rust | ✅ builds | ❌ no build |
| `java-spring-jooq-keycloak` | Java | ✅ builds | ❌ no build |
| `elixir-broadway-absinthe` | Elixir | ❌ no build | ✅ builds |
| `ai-search-workbench` | TypeScript | ❌ no build | ❌ no build |
| `go-realtime-api` | Go | ❌ no build | ❌ no build |
| `multi-dotnet-ops` | Multi (TS + .NET) | ❌ no build | ❌ no build |
| `ts-svelte-edge-orpc` | TypeScript | ❌ timeout | ❌ no build |
| `dotnet-blazor-cqrs` | .NET | ❌ no build | ❌ no build |
| `react-native-expo` | React Native | ❌ no build | ❌ no build |
| `frontier-effect-eventsourcing` | TypeScript | ❌ no build | ⚪ timeout |
| `frontier-polyglot-proto` | Multi | ⚪ inconclusive | ⚪ timeout |
| `multi-ts-go-grpc` | Multi (TS + Go) | ⚪ inconclusive | ⚪ inconclusive |

Opus uniquely lands `rust-leptos-axum` and `java-spring-jooq-keycloak`; **Sonnet 5 uniquely lands
`elixir-broadway-absinthe`** — the two flagship models have genuinely different strong suits. `python`
is the one spec both reliably build; the dense multi-service specs defeat both.

Sonnet 5 is also the **busiest** run in the field. It averaged **215 tool steps** per project — more
than double Opus at max (99) and ~6× Opus at low (38) — grinding through far more edits and retries,
and still landing fewer builds:

## Maximum reasoning didn't buy more builds

The obvious hypothesis after ScaffBench 2 was that the models knew the right libraries and just ran out
of room to wire them — so maximum reasoning should turn the near-misses into builds. It didn't.

On the same 13 specs, Opus 4.8 scored **identical at low and max — 3/11 either way** — for **5–10× the
cost**:

And the tie hides the most interesting result. Max didn't build the *same* three projects as low — it
**fixed two and broke two**:

| Spec | Opus 4.8 · low | Opus 4.8 · max |
| --- | :---: | :---: |
| `python-ingestion-api` | ✅ | ✅ |
| `rust-leptos-axum` | ❌ | ✅ **fixed** |
| `java-spring-jooq-keycloak` | ❌ | ✅ **fixed** |
| `go-realtime-api` | ✅ | ❌ **broke** |
| `multi-dotnet-ops` | ✅ | ❌ **broke** |

Extra deliberation **traded one project's success for another's** rather than lifting the total. More
thinking changed *which* hard stacks Opus assembled correctly, not *how many*. The bottleneck isn't
reasoning — it's coherent assembly.

## The wider field: Gemini 3.5 Flash, Sonnet 4.6, and Codex Spark

Three more configs ran at high effort — and the newcomer surprises.

- **Gemini 3.5 Flash ties Opus 4.8 for the top of the board at 3/11** — the strongest showing outside
  the Opus line, and from the field's lightest tier. It's also the **only model that built
  `dotnet-blazor-cqrs`**, and it cleared `java-spring` and `python`. Reached through **Google's
  Antigravity CLI** (`agy`) — the standalone Gemini CLI is now sunset for individual tiers — it
  finished all 13 with zero timeouts. Like Codex, its misses skew toward `install` failures.
  Antigravity emits plain text, so its cost / tokens / steps aren't measurable here.
- **GPT-5.3 Codex Spark is astonishingly terse: ~12 tool steps per project**, against Sonnet 5's 215
  and Opus's 38–99. It never spun — it finished all 13 specs, including both frontier ones, with zero
  timeouts. But that speed ships looser manifests: Codex fails mostly at **`install`** (its generated
  `package.json` / lockfiles don't resolve), whereas the Claude models fail at **`build` / `typecheck`**
  (they install fine, then don't compile). Two different ways to not-build.
- **Sonnet 4.6 at high** is the lightweight Claude: 46 steps, $1.53/project, 1/11 — the same
  python-only result as Codex, at a fraction of Sonnet 5's cost and step count.

The pattern holds across the vendor line: every model wires the right libraries (92–97%), and every
model struggles to assemble them into a build — whether it writes tersely (Codex) or grinds (Sonnet 5).

## The free tier: wired like the frontier, or barely there

We also ran four free models — two through **opencode** (DeepSeek V4 Flash, MiMo V2.5) and two through
**Kilo Code** (NVIDIA Nemotron 3 Nano 30B and Ultra 550B). They split into two very different stories.

| Free model | Backend | CORE pass@1 | Wired libs |
| --- | --- | :---: | :---: |
| DeepSeek V4 Flash | opencode | **1 / 10** (10%) | 83% |
| MiMo V2.5 | opencode | 0 / 9 (0%) | 82% |
| Nemotron 3 Nano 30B | Kilo | 0 / 1 | 7% |
| Nemotron 3 Ultra 550B | Kilo | 0 / 3 | 19% |

- **The opencode pair wire like the frontier but can't assemble it.** DeepSeek V4 Flash and MiMo V2.5
  both select ~82–83% of the right libraries — within reach of the paid field's 92–97% — yet land at
  **1/10 and 0/9 builds**. DeepSeek's single build (`python`, again) makes it the strongest free model on
  the board. Same failure shape as everyone else, just more of it: the stack is right, the assembly isn't.
- **The Kilo Nemotrons mostly don't produce a measurable project at all.** Nemotron 3 Nano 30B wired
  **7%** with 11 of 12 specs inconclusive; Ultra 550B wired **19%**. These read less like build failures
  than generation failures — the models research, stall, or emit too little to validate. Ultra 550B was
  slow enough that we stopped it at 10 of 13, and one spec whose build hung the validator is marked
  inconclusive rather than counted.

So the free floor isn't uniform. A capable open model can match the frontier at *choosing* the stack —
the whole gap is in *building* it — while a weaker one never gets far enough to have a build to fail.

## What's new in ScaffBench 2.1

- **Eight ecosystems, 13 specs.** TypeScript, Rust, Python, Go, .NET, and now **Java** (Spring + jOOQ +
  Keycloak) and **Elixir** (Broadway + Absinthe), plus an Expo React Native app and multi-ecosystem
  graphs — each chosen so a confident agent can pick a *near* neighbour and get it subtly wrong.
- **Frontier, prompt-only specs.** Two specs are deliberately beyond our own generator's option space,
  so they only ever run on the prompt path — no scaffolder could "cheat" them.
- **Real cost accounting.** Claude Code reports `$0` on a subscription, which made capable models look
  free. We now price every run from its token usage — which is how the max-effort cost above is even
  visible.
- **Validated end to end.** With the .NET SDK on the grading host, the .NET and multi-.NET specs are
  now genuinely built rather than skipped.

## The honest part

Three of Sonnet's specs and two of Opus's aren't clean build verdicts, and the benchmark is only useful
if we say so.

<Callout kind="warn">
  **`multi-ts-go-grpc` is inconclusive for both models — and it's our bug, not theirs.** Both agents
  built the project with manifests in subdirectories (`frontend/package.json`, `backend/go.mod`), but
  our validator only looks at the project root, so it ran zero build steps. The projects may well
  compile; the harness simply couldn't measure them, so they're excluded from every rate above.
  (Teaching the validator to find manifests in subdirectories is the next fix.)
</Callout>

The rest are timeouts. Opus 4.8 was cut off mid-reasoning on `ts-svelte-edge-orpc`; Sonnet 5 ran out
the clock on both **frontier** specs without producing a measurable project. Per our SWE-bench-style
taxonomy a generation timeout is a model failure, but it's a softer signal than a clean non-build, so
we surface it separately rather than folding it into the headline rate.

## How we score

<Callout kind="info">
  **CORE vs Full.** *CORE* is the load-bearing question — does the project install, build, type-check,
  and native-compile (`cargo check`, `go build`, `dotnet build`, `mvn`, `mix`)? *Full* adds a quality
  gate (lint, format, test). On the prompt path CORE is the headline — most runs don't get that far.
  Every run is `success`, `model-failure`, or `infra-inconclusive`; toolchain stalls aren't charged to
  the model, and (per SWE-bench) a generation timeout is.
</Callout>

## Takeaways

- **Sonnet 5 knows the stack, but assembles fewer builds.** It ties the field on library selection
  (~96% wired) yet trails on projects that actually compile (2/10 vs Opus's 3/11), while doing the most
  work to get there (215 steps/project). It does have its own strength — it's the only config that
  built the Elixir spec.
- **Reasoning effort isn't the lever.** Opus built the same 3/11 at max as at low, for 5–10× the cost —
  extra deliberation reshuffles which stacks assemble, not how many.
- **Stack selection is solved; coherent assembly isn't.** ~92–97% wired at a 9–27% build rate is the
  whole story in one line — true for every model in the field, and exactly the gap a scaffolder closes,
  which is why the MCP and CLI paths in ScaffBench 2 sat near 100% while prompt-only struggled.

We'll keep the leaderboard growing as more models land.
