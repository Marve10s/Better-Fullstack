当你想要 Django 的约定，同时希望项目形态以 API 为中心时，使用这个 stack。

```npm
npm create better-fullstack@latest my-django-api -- \
  --ecosystem python \
  --python-web-framework django \
  --python-orm sqlalchemy \
  --python-validation pydantic \
  --python-api django-rest-framework \
  --python-quality ruff
```

## 创建内容

- 使用 Django 的 Python 项目。
- Django REST Framework 作为 Python API 选项。
- SQLAlchemy 和 Pydantic 选项。
- Ruff 用于代码质量。

## 生成结构

这个 stack 面向想要 Django 生态，但需要 API-first 起点的开发者。生成的项目会记录 `--python-api django-rest-framework`，包含 Django 和 DRF 依赖，在应用入口中接入一个最小 Django settings 模块，并暴露一个类似 health check 的 API 响应。

```txt
my-django-api/
├── pyproject.toml
├── alembic.ini
├── migrations/
│   ├── env.py
│   └── versions/
├── src/app/
│   ├── main.py
│   ├── settings.py
│   ├── models.py
│   ├── schemas.py
│   ├── database.py
│   └── crud.py
└── tests/
    ├── test_main.py
    └── test_database.py
```

当你想在 Django HTTP 层之外保留明确的 Python 数据模型和服务层校验时，SQLAlchemy 与 Pydantic 仍然有价值。如果团队希望走更典型的 Django 路线，需要提前规划项目增长后是否把持久化迁移到 Django models 和 DRF serializers。

## 何时选择

如果约定、面向 admin 的结构和生态深度比极简 API 表面更重要，选择 Django。

如果你预计会添加 Django admin、Django auth、传统 middleware，或多个 Django apps，选择这个 stack 而不是 FastAPI 指南。如果服务应该保持小而专注于路由，并且容易和 AI 或数据处理代码放在一起，选择 FastAPI。

| 决策点 | Django REST API | FastAPI 服务 |
| --- | --- | --- |
| 团队预期 | Django 约定和生态 | 明确的 Python 模块和路由函数 |
| API 层 | Django REST Framework | FastAPI + Pydantic |
| Back-office/admin | 天然扩展点 | 添加单独工具 |
| Middleware/auth | Django 原生路径 | 显式组合 |
| 启动复杂度 | 更多 framework 结构 | 初始表面更小 |

## 代表性片段

生成的 DRF endpoint 会刻意保持很小。随着 API 增长，可以把 view functions 移到应用模块中继续扩展。

```python
from rest_framework.decorators import api_view
from rest_framework.response import Response


@api_view(["GET"])
def api_status(request):
    return Response({"status": "ok", "framework": "django-rest-framework"})
```

对于实际资源 endpoint，即使继续使用 SQLAlchemy 作为持久化工具，也建议添加一个类似 serializer 的边界。

```python
from pydantic import BaseModel, EmailStr
from rest_framework.decorators import api_view
from rest_framework.response import Response


class UserCreate(BaseModel):
    email: EmailStr
    name: str | None = None


@api_view(["POST"])
def create_user(request):
    payload = UserCreate.model_validate(request.data)
    user = user_service.create_user(email=payload.email, name=payload.name)
    return Response({"id": user.id, "email": user.email}, status=201)
```

如果之后转向更典型的 DRF serializers，也保留同样的边界：校验请求数据，调用应用逻辑，并返回稳定的响应形状。

```python
from rest_framework import serializers


class UserSerializer(serializers.Serializer):
    id = serializers.IntegerField(read_only=True)
    email = serializers.EmailField()
    name = serializers.CharField(required=False, allow_blank=True)
```

## 迁移与测试

如果采用 Django models，就使用 Django migrations。如果继续让 SQLAlchemy 作为持久化的事实来源，就使用 Alembic。除非 ownership 划分非常清楚，否则不要让两个迁移系统管理同一批表。

```bash
uv run ruff check .
uv run pytest
```

在传统 Django 项目 layout 中使用 Django-model 持久化时：

```bash
uv run python manage.py makemigrations
uv run python manage.py migrate
```

使用 SQLAlchemy 持久化时：

```bash
uv run alembic revision --autogenerate -m "add users"
uv run alembic upgrade head
```

路由测试应该使用 Django test client 或 DRF API test client，并断言 JSON 契约，而不仅是 view 实现细节。

## 兼容性说明

- `--python-api django-rest-framework` 需要 `--python-web-framework django`。
- 不要把 `--python-api django-rest-framework` 加到 FastAPI stack；CLI validation 会拒绝这个组合。
- `--python-orm sqlalchemy` 在 Python 生态中可用，但 Django 内置 admin 和 generic views 是围绕 Django models 设计的。
- `--python-validation pydantic` 适合服务层 payload 校验，而 DRF serializers 是 DRF views 中传统的 request/response 工具。

## 部署说明

Django 部署需要常规的生产 settings 检查：配置 `SECRET_KEY`、`ALLOWED_HOSTS`、trusted origins、数据库凭据和静态资源处理。如果生成的 app 仍然很小，把它当作起点，在生产前把 settings 移到明确的、由环境驱动的模块中。

```bash
HOST=0.0.0.0 PORT="${PORT:-8000}" uv run python -m app.main
```

如果你继续使用生成的轻量 entrypoint，而不是完整 Django project layout，请记录 settings 从哪里加载，避免后续 Django 开发者误以为默认的 `settings.py` 和 `manage.py` 结构完整存在。生产前，还要添加平台期望的 WSGI/ASGI server 与部署 settings。

## 故障排查

| 症状 | 检查 |
| --- | --- |
| DRF imports 失败 | 确认项目是用 `--python-api django-rest-framework` 创建的。 |
| Django settings 未配置 | 确保在导入 Django models 或 views 前设置了 `DJANGO_SETTINGS_MODULE`。 |
| 迁移 ownership 混乱 | 为每张表选择 Django migrations 或 Alembic 作为 owner。 |
| FastAPI 示例不适用 | 这个 stack 使用 Django/DRF routing，而不是 FastAPI decorators。 |

## 取舍

如果想要更小的 API 服务，使用 [FastAPI with PostgreSQL and SQLAlchemy](/guides/python/fastapi-postgres-sqlalchemy/)。如果需要 Django 约定，这个 stack 更合适。
