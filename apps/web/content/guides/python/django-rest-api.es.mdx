Usa este stack cuando quieras las convenciones de Django con una forma de proyecto centrada en APIs.

```npm
npm create better-fullstack@latest my-django-api -- \
  --ecosystem python \
  --python-web-framework django \
  --python-orm sqlalchemy \
  --python-validation pydantic \
  --python-api django-rest-framework \
  --python-quality ruff
```

## Qué crea

- Un proyecto Python con Django.
- Django REST Framework como opción de API para Python.
- Selecciones de SQLAlchemy y Pydantic.
- Ruff para calidad de código.

## Forma generada

Este stack está pensado para equipos que quieren el ecosistema Django, pero con un punto de partida API-first. El proyecto generado registra `--python-api django-rest-framework`, incluye dependencias de Django y DRF, configura un módulo mínimo de settings dentro del entrypoint de la app y expone una respuesta de API tipo health check.

```txt
my-django-api/
├── pyproject.toml
├── alembic.ini
├── migrations/
│   ├── env.py
│   └── versions/
├── src/app/
│   ├── main.py
│   ├── settings.py
│   ├── models.py
│   ├── schemas.py
│   ├── database.py
│   └── crud.py
└── tests/
    ├── test_main.py
    └── test_database.py
```

Las selecciones de SQLAlchemy y Pydantic siguen siendo útiles cuando quieres modelos de datos explícitos y validación en una capa de servicios alrededor de una superficie HTTP de Django. Si tu equipo prefiere el camino más idiomático de Django, decide cuándo conviene migrar la persistencia hacia modelos de Django y serializers de DRF a medida que el proyecto crezca.

## Cuándo elegirlo

Elige Django cuando las convenciones, una estructura orientada a administración y la profundidad del ecosistema importen más que una superficie de API mínima.

Elige este stack antes que el de FastAPI si esperas agregar Django admin, autenticación de Django, middleware convencional o varias apps de Django. Elige FastAPI cuando el servicio deba mantenerse pequeño, enfocado en rutas y fácil de integrar junto a código de AI o procesamiento de datos.

| Punto de decisión | Django REST API | Servicio FastAPI |
| --- | --- | --- |
| Expectativa del equipo | Convenciones y ecosistema Django | Módulos Python y funciones de ruta explícitos |
| Capa de API | Django REST Framework | FastAPI + Pydantic |
| Back-office/admin | Punto de extensión natural | Agregar una herramienta separada |
| Middleware/auth | Camino nativo de Django | Composición explícita |
| Complejidad inicial | Más estructura de framework | Superficie inicial más pequeña |

## Fragmentos representativos

El endpoint DRF generado es intencionalmente pequeño. Expándelo moviendo las funciones de vista a módulos de la app cuando la API crezca.

```python
from rest_framework.decorators import api_view
from rest_framework.response import Response


@api_view(["GET"])
def api_status(request):
    return Response({"status": "ok", "framework": "django-rest-framework"})
```

Para un endpoint de recurso real, agrega un límite tipo serializer aunque mantengas SQLAlchemy como herramienta de persistencia.

```python
from pydantic import BaseModel, EmailStr
from rest_framework.decorators import api_view
from rest_framework.response import Response


class UserCreate(BaseModel):
    email: EmailStr
    name: str | None = None


@api_view(["POST"])
def create_user(request):
    payload = UserCreate.model_validate(request.data)
    user = user_service.create_user(email=payload.email, name=payload.name)
    return Response({"id": user.id, "email": user.email}, status=201)
```

Si más adelante te mueves hacia serializers idiomáticos de DRF, conserva el mismo límite: validar datos de request, llamar lógica de aplicación y devolver una forma de respuesta estable.

```python
from rest_framework import serializers


class UserSerializer(serializers.Serializer):
    id = serializers.IntegerField(read_only=True)
    email = serializers.EmailField()
    name = serializers.CharField(required=False, allow_blank=True)
```

## Migraciones y pruebas

Usa migraciones de Django si adoptas modelos de Django. Usa Alembic si mantienes SQLAlchemy como fuente de verdad para la persistencia. Evita ejecutar ambos sistemas de migración contra las mismas tablas salvo que tengas una división de ownership muy clara.

```bash
uv run ruff check .
uv run pytest
```

Para persistencia con modelos de Django en un layout convencional de proyecto Django:

```bash
uv run python manage.py makemigrations
uv run python manage.py migrate
```

Para persistencia con SQLAlchemy:

```bash
uv run alembic revision --autogenerate -m "add users"
uv run alembic upgrade head
```

Las pruebas de rutas deberían usar el test client de Django o el API test client de DRF y validar el contrato JSON, no solo detalles internos de la vista.

## Notas de compatibilidad

- `--python-api django-rest-framework` requiere `--python-web-framework django`.
- No agregues `--python-api django-rest-framework` a un stack FastAPI; la validación de la CLI rechaza esa combinación.
- `--python-orm sqlalchemy` está disponible en el ecosistema Python, pero el admin y las generic views de Django están diseñados alrededor de modelos de Django.
- `--python-validation pydantic` es útil para validar payloads en la capa de servicios, mientras que los serializers de DRF son la herramienta convencional de request/response dentro de vistas DRF.

## Notas de despliegue

Los despliegues de Django necesitan la pasada habitual de settings de producción: configura `SECRET_KEY`, `ALLOWED_HOSTS`, orígenes confiables, credenciales de base de datos y manejo de assets estáticos. Si la app generada todavía es mínima, trátala como punto de partida y mueve los settings a módulos explícitos basados en entorno antes de producción.

```bash
HOST=0.0.0.0 PORT="${PORT:-8000}" uv run python -m app.main
```

Si mantienes el entrypoint ligero generado en lugar de un layout Django completo, documenta dónde se cargan los settings para que futuros colaboradores de Django no asuman que la estructura estándar de `settings.py` y `manage.py` existe sin cambios. Antes de producción, agrega el servidor WSGI/ASGI y los settings de despliegue que tu plataforma espere.

## Troubleshooting

| Síntoma | Revisa |
| --- | --- |
| Fallan los imports de DRF | Confirma que el proyecto se creó con `--python-api django-rest-framework`. |
| Los settings de Django no están configurados | Asegúrate de definir `DJANGO_SETTINGS_MODULE` antes de importar modelos o vistas de Django. |
| Confusión con migraciones | Elige Django migrations o Alembic como dueño de cada tabla. |
| Los ejemplos de FastAPI no aplican | Este stack usa routing de Django/DRF, no decoradores de FastAPI. |

## Tradeoffs

Para un servicio de API más pequeño, usa [FastAPI with PostgreSQL and SQLAlchemy](/guides/python/fastapi-postgres-sqlalchemy/). Para convenciones Django, este stack encaja mejor.
