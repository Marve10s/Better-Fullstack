Usa este stack cuando quieras un servidor API TypeScript rápido y un frontend React separado con llamadas cliente-servidor tipadas.

```npm
npm create better-fullstack@latest my-hono-app -- \
  --ecosystem typescript \
  --frontend tanstack-router \
  --backend hono \
  --runtime bun \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## Qué crea

- Un frontend TanStack Router.
- Un backend Hono ejecutándose en Bun.
- PostgreSQL y Drizzle.
- Better Auth.
- tRPC entre la app web y la API.
- Una estructura monorepo para paquetes de frontend y servidor.

## Forma generada

Este comando crea una forma de app dividida: frontend y backend son paquetes separados, pero se generan juntos y comparten la misma configuración de stack. Eso hace explícito el límite de API sin perder inferencia TypeScript.

Forma representativa:

```txt
my-hono-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

Los nombres exactos de workspace pueden variar, pero el punto arquitectónico es el mismo: TanStack Router es dueño de la app del navegador, Hono de HTTP, Drizzle del acceso a datos y tRPC del contrato tipado entre ambos.

## Ruta Hono de ejemplo

Hono es útil cuando el manejo de requests debe seguir siendo explícito y liviano:

```ts
import { Hono } from "hono";

const app = new Hono();

app.get("/health", (c) => {
  return c.json({ ok: true });
});

export default app;
```

Mantén middleware de framework, contexto de auth y logging de requests en el paquete server. Evita filtrar objetos request de Hono hacia la lógica de negocio compartida.

## Forma de procedimiento tRPC de ejemplo

Los procedimientos tRPC deben ser puntos de entrada pequeños y validados alrededor de operaciones de servidor:

```ts
import { z } from "zod";

const createTaskInput = z.object({
  title: z.string().min(1),
});

export const taskProcedures = {
  create: async (rawInput: unknown, userId: string) => {
    const input = createTaskInput.parse(rawInput);

    return {
      id: crypto.randomUUID(),
      title: input.title,
      ownerId: userId,
    };
  },
};
```

En apps grandes, pon schemas de validación en un módulo compartido solo cuando cliente y servidor los necesiten. Las escrituras de base de datos deben quedarse del lado servidor.

## Cuándo elegirlo

Elige Hono cuando el backend deba mantenerse pequeño, rápido y explícito. Esta configuración funciona bien para productos API-first, dashboards, herramientas internas y servicios que luego podrían separar despliegues de frontend y backend.

## Compatibilidad

- `--frontend tanstack-router` más `--backend hono` da una app web React separada y un servidor API.
- `--runtime bun` selecciona Bun para el servicio Hono en esta guía.
- `--api trpc` encaja aquí porque el frontend se basa en React. Para guías SvelteKit, Nuxt o SolidStart, Better Fullstack usa oRPC.
- PostgreSQL y Drizzle son una pareja orientada a producción para esta forma de servicio dividido.

## Despliegue

Planifica dos objetivos desplegables si tu hosting trata el frontend y el servidor Hono por separado. Configura CORS, dominios de cookies de auth y URLs públicas de API de forma deliberada; los servicios divididos hacen visibles esos detalles.

El servidor API necesita la cadena PostgreSQL y secretos de auth. La app web necesita el origen de la API y cualquier valor público de URL de app. Mantén secretos fuera de variables públicas de build del frontend.

## Solucion de problemas

- Si las requests del navegador fallan antes de llegar a Hono, inspecciona CORS y la URL base de API configurada.
- Si faltan cookies de auth entre dominios, revisa same-site, secure cookies y si los dominios web/API coinciden con tu despliegue previsto.
- Si las llamadas tRPC compilan pero fallan en runtime, verifica que el adapter tRPC de Hono esté montado en la misma ruta usada por el cliente.
- Si las apps generadas no pueden conectarse a PostgreSQL localmente, confirma que la base esté corriendo y que la cadena de conexión coincida con la base seleccionada.

## Tradeoffs

Si quieres que un solo framework sea dueño de toda la app, usa [TanStack Start](/guides/typescript/create-tanstack-start-project/) o [Next.js](/guides/typescript/nextjs-drizzle-better-auth/). Si quieres un límite API separado, Hono es el valor por defecto más claro.

Comparado con un framework fullstack con self-backend, este stack tiene más piezas móviles. La recompensa es un límite de servicio más limpio, testing API-focused más sencillo y un backend que luego puede servir a varios clientes.
