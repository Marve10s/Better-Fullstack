Usa este stack cuando quieras una app React fullstack mainstream con datos relacionales, autenticación y una capa de API tipada.

```npm
npm create better-fullstack@latest my-next-app -- \
  --ecosystem typescript \
  --frontend next \
  --backend self \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## Qué crea

- Una app Next.js en modo fullstack.
- PostgreSQL como base de datos relacional.
- Drizzle como ORM TypeScript.
- Better Auth para autenticación.
- tRPC para APIs tipadas end-to-end.
- Tailwind CSS y shadcn/ui para estilos frontend.

## Forma generada

La app generada usa Next.js como límite web y servidor. PostgreSQL y Drizzle quedan detrás de módulos server-only, mientras tRPC ofrece llamadas tipadas para componentes cliente que necesitan datos de aplicación.

Forma representativa:

```txt
my-next-app/
  bts.jsonc
  app/
  components/
  lib/
    auth/
    db/
    trpc/
```

Las rutas exactas pueden variar según la versión del generator, pero la separación prevista es consistente: los componentes UI renderizan la experiencia de producto, route handlers y código de servidor son dueños de los límites request, y el acceso a base de datos no entra en bundles de navegador.

## Modelo Drizzle de ejemplo

Mantén las tablas de producto explícitas y fáciles de migrar:

```ts
import { pgTable, text, timestamp } from "drizzle-orm/pg-core";

export const workspace = pgTable("workspace", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  ownerId: text("owner_id").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

Better Auth debe ser dueño de las tablas de autenticación y el comportamiento de sesión. Las tablas de producto deben referenciar IDs de usuario y evitar copiar campos de sesión dentro de modelos de aplicación.

## Procedimiento tipado de ejemplo

Usa tRPC para operaciones donde el frontend se beneficia de tipos inferidos de entrada y salida:

```ts
import { z } from "zod";

export const createWorkspaceInput = z.object({
  name: z.string().min(2).max(80),
});

export async function createWorkspace(rawInput: unknown, userId: string) {
  const input = createWorkspaceInput.parse(rawInput);

  return {
    id: crypto.randomUUID(),
    name: input.name,
    ownerId: userId,
  };
}
```

Mantén la comprobación de autorización cerca de la mutación. Crear un workspace debe demostrar que hay un usuario autenticado antes de insertar nada.

## Cuándo elegirlo

Elige esto para apps SaaS, productos con cuentas, dashboards, portales de administración y proyectos donde la familiaridad de hosting de Next.js importa.

## Por que Drizzle y Better Auth

Drizzle mantiene el acceso a base de datos cerca de TypeScript. Better Auth aporta una capa moderna de auth sin obligar a usar un proveedor de identidad hosted por defecto.

## Compatibilidad

- `--backend self` significa que Next.js es dueño del límite de servidor; no agregues un backend separado salvo que quieras una división explícita de servicios.
- `--database postgres` encaja bien con hosting de producción, conexiones pooled y proveedores de base de datos gestionados.
- `--api trpc` es una buena opción para este stack basado en React. Los frontends TypeScript no React en Better Fullstack suelen usar oRPC.
- shadcn/ui espera Tailwind y React. Mantén esas decisiones alineadas si luego regeneras o agregas componentes.

## Despliegue

Configura la cadena PostgreSQL, los secretos de Better Auth y la URL base de la app en el entorno de despliegue. Ejecuta migraciones antes de enviar tráfico de producción a código que espera columnas o tablas nuevas.

En hosting serverless, revisa la recomendación de pooling del proveedor de base de datos. Las conexiones directas a PostgreSQL pueden agotar límites rápido cuando arrancan muchas instancias serverless a la vez.

## Solucion de problemas

- Si auth funciona localmente pero no en producción, verifica callback URLs, trusted origins, cookies y la app URL de producción.
- Si las migraciones corren pero la app sigue viendo errores de schema antiguo, confirma que la app y el comando de migración apunten a la misma base.
- Si componentes cliente importan módulos server-only de base de datos, mueve la query detrás de un route handler, server action o procedimiento API tipado.
- Si los tipos tRPC no se actualizan, reinicia el servidor TypeScript y vuelve a correr el typecheck específico de la app.
