Usa este stack cuando quieras Astro para páginas de contenido e islands, junto con un servidor API TypeScript.

```npm
npm create better-fullstack@latest my-astro-app -- \
  --ecosystem typescript \
  --frontend astro \
  --astro-integration react \
  --backend hono \
  --runtime bun \
  --database sqlite \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --css-framework tailwind \
  --ui-library daisyui \
```

## Qué crea

- Un frontend Astro con integración React.
- Un backend Hono ejecutándose en Bun.
- SQLite y Drizzle.
- Better Auth.
- tRPC, porque esta configuración de Astro usa integración React.
- Tailwind CSS y DaisyUI.

## Forma generada

Este stack combina el frontend content-first de Astro con un servidor API Hono separado. Las React islands se activan mediante `--astro-integration react`, por eso tRPC está disponible en esta guía.

Forma representativa:

```txt
my-astro-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

Las páginas Astro deben centrarse en routing, contenido y composición de islands. Hono debe ser dueño de las rutas API, el comportamiento de servidor sensible a auth y el acceso a base de datos.

## Astro island de ejemplo

Usa Astro para la carcasa de la página y React solo donde haga falta interactividad:

```astro
---
import ProjectPicker from "../components/ProjectPicker.tsx";
---

<main class="mx-auto max-w-5xl p-6">
  <h1 class="text-3xl font-semibold">Project docs</h1>
  <ProjectPicker client:load />
</main>
```

Así las páginas estáticas o casi estáticas siguen siendo baratas, mientras los componentes React interactivos pueden llamar a la API tipada.

## Endpoint Hono de ejemplo

Los health checks y endpoints de servicio simples pertenecen en Hono:

```ts
import { Hono } from "hono";

const app = new Hono();

app.get("/health", (c) => c.json({ ok: true }));

export default app;
```

Para endpoints con base de datos, mantén las queries Drizzle detrás de helpers server-only y expónlas mediante procedimientos tRPC o rutas Hono.

## Compatibilidad

La compatibilidad de API en Astro depende de la integración UI seleccionada. Esta página usa `--astro-integration react`, lo que mantiene tRPC disponible. Para integraciones Astro que no sean React, prefiere oRPC.

DaisyUI se selecciona como librería UI en este comando. Si luego cambias a shadcn/ui, asegúrate de que la integración React de Astro y Tailwind sigan alineadas con la librería de componentes elegida.

## Cuándo elegirlo

Elige Astro cuando el contenido, docs, páginas de marketing o partial hydration sean centrales para la app, pero todavía necesites backend generado y base de datos.

## Despliegue

Planifica orígenes claros para el frontend Astro y la API Hono. El frontend necesita la URL pública de la API. El servicio Hono necesita secretos de base de datos y auth. SQLite puede servir para desarrollo local, pero el hosting de producción debe ofrecer almacenamiento persistente si mantienes SQLite.

Si frontend y API se despliegan por separado, configura CORS y cookies de auth para los dominios finales en vez de depender del comportamiento de localhost.

## Solucion de problemas

- Si las React islands no hidratan, revisa la directiva client de Astro y si el componente importa código browser-only de forma segura.
- Si tRPC deja de estar disponible tras cambiar integraciones, confirma que la integración de Astro sigue siendo React.
- Si las llamadas API fallan desde páginas desplegadas, verifica el origen público de la API y la configuración de CORS.
- Si los datos SQLite desaparecen tras desplegar, confirma que tu plataforma de hosting soporta almacenamiento escribible persistente.

## Notas de comparacion

Elige esto cuando las páginas de contenido y la interactividad selectiva importen más que una carcasa React completa. Elige [Hono con tRPC y Drizzle](/guides/typescript/hono-trpc-drizzle/) cuando el frontend deba ser una app React dedicada con TanStack Router.
