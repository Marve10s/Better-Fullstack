Usa este stack cuando quieras una app fullstack Vue/Nuxt con datos relacionales, autenticación y contratos de API tipados.

```npm
npm create better-fullstack@latest my-nuxt-app -- \
  --ecosystem typescript \
  --frontend nuxt \
  --backend self \
  --database postgres \
  --orm prisma \
  --auth better-auth \
  --api orpc \
  --css-framework tailwind \
```

## Qué crea

- Una app Nuxt en modo fullstack.
- PostgreSQL y Prisma.
- Better Auth.
- oRPC para contratos de API tipados.
- Tailwind CSS.
- Una configuración reproducible en `bts.jsonc`.

## Forma generada

Este stack mantiene la aplicación dentro del modelo fullstack de Nuxt. Prisma es dueño del schema y las migraciones de base de datos, Better Auth del comportamiento de autenticación, y oRPC aporta contratos tipados sin asumir un frontend React.

Forma representativa:

```txt
my-nuxt-app/
  bts.jsonc
  app/
  server/
  prisma/
    schema.prisma
```

Tómalo como una distribución representativa. El límite importante es que el código server-only de base de datos y auth permanece en módulos de servidor Nuxt o handlers API, mientras los componentes Vue llaman contratos tipados de servidor.

## Modelo Prisma de ejemplo

Un modelo pequeño de producto puede empezar así:

```prisma
model Workspace {
  id        String   @id
  name      String
  ownerId   String
  createdAt DateTime @default(now())
}
```

Mantén los modelos requeridos por Better Auth alineados con la documentación del adapter y separados de las tablas específicas del producto. Las tablas de producto deben referenciar usuarios por identificador, no duplicar datos de login.

## Forma de contrato oRPC de ejemplo

Usa oRPC para describir operaciones de servidor validadas para la app Nuxt:

```ts
import { z } from "zod";

export const createWorkspaceInput = z.object({
  name: z.string().min(2),
});

export async function createWorkspace(rawInput: unknown, userId: string) {
  const input = createWorkspaceInput.parse(rawInput);

  return {
    id: crypto.randomUUID(),
    name: input.name,
    ownerId: userId,
  };
}
```

Pon las comprobaciones de autorización en la implementación del servidor, no solo en componentes Vue. Los guards del cliente mejoran la UX, pero no protegen los datos.

## Compatibilidad

Nuxt usa oRPC en Better Fullstack para contratos tipados. tRPC se reserva para frontends TypeScript basados en React, por eso esta guía usa `--api orpc`.

`--backend self` es la elección correcta para Nuxt fullstack. Si quieres un servidor API desplegado por separado, usa una opción de backend como Hono en un stack que soporte esa combinación.

## Cuándo elegirlo

Elige Nuxt cuando tu equipo prefiera Vue, quiera routing con capacidades de servidor y necesite una forma fullstack en vez de frontend y API server separados.

## Despliegue

Configura la cadena de conexión PostgreSQL, los secretos de Better Auth y la URL pública de la app en el entorno de despliegue. Ejecuta migraciones Prisma antes de desplegar código que dependa de schema nuevo.

En plataformas serverless, revisa cómo se genera Prisma Client y cómo se gestionan los pools de conexión. Los proveedores PostgreSQL gestionados suelen documentar la cadena de conexión más segura para runtimes serverless.

## Solucion de problemas

- Si los tipos de Prisma Client están obsoletos, regenera el cliente después de cambiar `schema.prisma`.
- Si las llamadas oRPC fallan solo en producción, verifica la URL base de rutas de servidor y el comportamiento del adapter de despliegue.
- Si los redirects de auth entran en bucle, revisa trusted origins, cookies y la app URL usada por Better Auth.
- Si las migraciones apuntan a la base incorrecta, inspecciona la variable de entorno usada tanto por comandos locales como por el runtime desplegado.
