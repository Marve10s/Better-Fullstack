Usa este stack cuando quieras la reactividad fina de Solid con un framework fullstack y contratos de API tipados.

```npm
npm create better-fullstack@latest my-solidstart-app -- \
  --ecosystem typescript \
  --frontend solid-start \
  --backend self \
  --database sqlite \
  --orm drizzle \
  --auth better-auth \
  --api orpc \
  --css-framework tailwind \
```

## Qué crea

- Una app SolidStart fullstack.
- SQLite y Drizzle.
- Better Auth.
- oRPC para contratos tipados.
- Tailwind CSS.

## Forma generada

Este stack mantiene SolidStart como framework fullstack. La app puede renderizar UI Solid, llamar contratos tipados de servidor y persistir datos con Drizzle sin introducir un servicio backend separado.

Forma representativa:

```txt
my-solidstart-app/
  bts.jsonc
  src/
    routes/
    components/
    lib/
      auth/
      db/
      rpc/
```

Usa este árbol como mapa de ownership. El código UI pertenece a componentes y rutas Solid; el código de base de datos y auth debe permanecer server-only.

## Componente de ruta de ejemplo

Las rutas SolidStart pueden centrarse en renderizar mientras el código de servidor maneja acceso a datos:

```tsx
export default function ProjectsPage() {
  return (
    <main class="mx-auto max-w-4xl p-6">
      <h1 class="text-2xl font-semibold">Projects</h1>
      <p class="text-gray-600">Create typed server calls for project data.</p>
    </main>
  );
}
```

Mantén las mutaciones de servidor detrás de funciones validadas para que la UI no sea la única fuente de verdad para las reglas de input.

## Modelo Drizzle de ejemplo

SQLite es un valor por defecto práctico para apps SolidStart pequeñas y desarrollo local:

```ts
import { integer, sqliteTable, text } from "drizzle-orm/sqlite-core";

export const note = sqliteTable("note", {
  id: text("id").primaryKey(),
  body: text("body").notNull(),
  ownerId: text("owner_id").notNull(),
  createdAt: integer("created_at", { mode: "timestamp" }).notNull(),
});
```

Pásate a PostgreSQL cuando varias instancias desplegadas necesiten compartir datos relacionales durables.

## Compatibilidad

SolidStart usa oRPC en Better Fullstack. tRPC está orientado a React, por eso este stack mantiene `--api orpc`.

`--backend self` significa que SolidStart es dueño de la capa servidor. No agregues Hono salvo que quieras intencionalmente un despliegue API separado.

## Cuándo elegirlo

Elige SolidStart cuando quieras el modelo runtime de Solid pero también necesites un proyecto fullstack generado con base de datos, auth y wiring de API tipada.

## Despliegue

Configura los secretos de Better Auth, la app URL y la ruta o cadena de conexión de base de datos para el entorno destino. Si tu plataforma de despliegue no ofrece disco persistente, no dependas de SQLite basado en archivo para datos de producción.

Para herramientas internas pequeñas, SQLite puede ser suficiente. Para apps de producción multiusuario tipo SaaS, PostgreSQL suele ser más fácil de operar entre despliegues.

## Solucion de problemas

- Si las llamadas oRPC fallan tras desplegar, revisa la ruta base y si el adapter de despliegue maneja rutas de servidor como esperas.
- Si las sesiones de auth desaparecen, revisa seguridad de cookies y valores de app URL.
- Si los datos SQLite se resetean, confirma si el host ofrece almacenamiento persistente.
- Si componentes Solid importan módulos de servidor por accidente, mueve la llamada detrás de una server function o contrato API.
