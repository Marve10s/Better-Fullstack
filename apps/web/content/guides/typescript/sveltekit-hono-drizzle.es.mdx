Usa este stack cuando quieras SvelteKit en el frontend y un pequeño servidor API TypeScript junto a él.

```npm
npm create better-fullstack@latest my-sveltekit-app -- \
  --ecosystem typescript \
  --frontend svelte \
  --backend hono \
  --runtime bun \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api orpc \
  --css-framework tailwind \
```

## Qué crea

- Un frontend SvelteKit.
- Un backend Hono en Bun.
- oRPC para contratos de API tipados entre opciones frontend no React.
- PostgreSQL y Drizzle.
- Better Auth cuando la base de datos y ORM seleccionados lo soportan.

## Forma generada

Este es un stack TypeScript dividido: SvelteKit maneja la experiencia frontend, Hono maneja el servidor API, y oRPC es el límite tipado entre ambos.

Forma representativa:

```txt
my-sveltekit-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

Los nombres específicos de workspace pueden evolucionar, pero mantén clara la propiedad. Los componentes Svelte no deben importar módulos de base de datos directamente; deben llamar operaciones tipadas de servidor.

## Patron de llamada SvelteKit de ejemplo

Una página o función load debe hablar con el cliente API generado en vez de hardcodear payloads de request por toda la app:

```ts
export async function load() {
  return {
    title: "Projects",
  };
}
```

Para mutaciones, mantén validación y autorización del lado Hono aunque la UI SvelteKit ya valide inputs de formulario.

## Modelo Drizzle de ejemplo

PostgreSQL más Drizzle te da definiciones explícitas de tablas:

```ts
import { pgTable, text, timestamp } from "drizzle-orm/pg-core";

export const project = pgTable("project", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  ownerId: text("owner_id").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

Usa funciones pequeñas de servidor alrededor de queries para que handlers Hono y procedimientos oRPC no se conviertan en un cajón de reglas de negocio.

## Cuándo elegirlo

Elige esto cuando tu equipo prefiera SvelteKit pero quiera un servidor API explícito y contratos tipados. Es útil para apps que pueden crecer hacia despliegues separados de frontend y backend.

## Compatibilidad

- Esta guía usa `--api orpc` porque SvelteKit no es un frontend React.
- `--backend hono` y `--runtime bun` crean un servicio API separado en vez de depender solo de rutas de servidor SvelteKit.
- Better Auth requiere que la base de datos y la ruta ORM elegidas sigan siendo compatibles. Mantén cambios de schema de auth y migraciones Drizzle sincronizados.
- Si luego eliminas el backend separado, revisa juntos el setup del cliente API, CORS y el manejo de cookies de auth.

## Despliegue

Espera variables de entorno separadas para web y API. La app SvelteKit necesita el origen público de la API. El servidor Hono necesita PostgreSQL y secretos de auth. Si se despliegan en dominios distintos, configura CORS y cookies intencionalmente.

Ejecuta migraciones de base de datos antes de desplegar código API que espera tablas o columnas nuevas.

## Solucion de problemas

- Si las requests fallan desde el navegador pero funcionan con curl, revisa CORS y credenciales de cookies.
- Si SvelteKit renderiza sin datos, verifica el origen API usado por código server-side y browser-side.
- Si auth es inconsistente entre dominios, revisa domain, same-site y secure en cookies.
- Si los cambios de schema Drizzle no aparecen, confirma que las migraciones se generaron y aplicaron a la base PostgreSQL correcta.

## Notas de comparacion

Comparado con un stack Nuxt o SolidStart self-backend, esta guía te da un límite API más explícito. Comparado con [Hono con tRPC y Drizzle](/guides/typescript/hono-trpc-drizzle/), usa oRPC porque el frontend es SvelteKit y no React.
