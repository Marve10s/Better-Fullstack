当你想在 frontend 使用 SvelteKit，并在旁边放一个小型 TypeScript API server 时，使用这个 stack。

```npm
npm create better-fullstack@latest my-sveltekit-app -- \
  --ecosystem typescript \
  --frontend svelte \
  --backend hono \
  --runtime bun \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api orpc \
  --css-framework tailwind \
```

## 创建内容

- 一个 SvelteKit frontend。
- Bun 上的 Hono backend。
- oRPC，用于非 React frontend choices 之间的 typed API contracts。
- PostgreSQL 和 Drizzle。
- 在所选 database 和 ORM 支持时启用 Better Auth。

## 生成结构

这是一个 split TypeScript stack：SvelteKit 处理 frontend experience，Hono 处理 API server，oRPC 是它们之间的 typed boundary。

代表性结构：

```txt
my-sveltekit-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

具体 workspace 名称可能演进，但职责要清晰。Svelte components 不应直接导入 database modules；它们应调用 typed server operations。

## 示例 SvelteKit call pattern

页面或 load function 应该与生成的 API client 通信，而不是在整个 app 中硬编码 request payloads：

```ts
export async function load() {
  return {
    title: "Projects",
  };
}
```

对于 mutations，即使 SvelteKit UI 已经验证 form inputs，也要把 validation 和 authorization 保持在 Hono 侧。

## 示例 Drizzle model

PostgreSQL 加 Drizzle 给你明确的 table definitions：

```ts
import { pgTable, text, timestamp } from "drizzle-orm/pg-core";

export const project = pgTable("project", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  ownerId: text("owner_id").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

围绕 queries 使用小型 server functions，避免 Hono handlers 和 oRPC procedures 变成 business rules 的堆放处。

## 何时选择

当团队偏好 SvelteKit，但仍想要明确 API server 和 typed contracts 时，选择这个 stack。它适合未来可能拆成独立 frontend/backend deployments 的 apps。

## 兼容性

- 本指南使用 `--api orpc`，因为 SvelteKit 不是 React frontend。
- `--backend hono` 和 `--runtime bun` 会创建独立 API service，而不是只依赖 SvelteKit server routes。
- Better Auth 要求所选 database 和 ORM 路径保持兼容。保持 auth schema changes 和 Drizzle migrations 同步。
- 如果之后移除独立 backend，需要一起重新检查 API client setup、CORS 和 auth cookie handling。

## 部署

预计 web 和 API 会有不同 environment variables。SvelteKit app 需要 public API origin；Hono server 需要 PostgreSQL 和 auth secrets。如果它们部署在不同 domains，请有意配置 CORS 和 cookies。

在部署依赖新 tables 或 columns 的 API code 前，先运行 database migrations。

## 故障排查

- 如果 requests 从 browser 失败但 curl 可用，检查 CORS 和 cookie credentials。
- 如果 SvelteKit render 没有数据，确认 server-side 和 browser-side code 使用的 API origin。
- 如果 auth 在跨域时不一致，检查 cookie domain、same-site 和 secure settings。
- 如果 Drizzle schema changes 没出现，确认 migrations 已生成并应用到目标 PostgreSQL database。

## 对比说明

相比 Nuxt 或 SolidStart self-backend stack，本指南提供更明确的 API boundary。相比 [Hono with tRPC and Drizzle](/guides/typescript/hono-trpc-drizzle/)，它使用 oRPC，因为 frontend 是 SvelteKit 而不是 React。
