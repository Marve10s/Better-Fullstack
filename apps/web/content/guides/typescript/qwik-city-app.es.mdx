Usa este stack cuando quieras una app Qwik sin agregar un backend ni una base de datos separada desde el primer día.

```npm
npm create better-fullstack@latest my-qwik-app -- \
  --ecosystem typescript \
  --frontend qwik \
  --backend none \
  --runtime none \
  --database none \
  --orm none \
  --auth none \
  --api none \
  --css-framework tailwind \
```

## Qué crea

- Una app Qwik.
- Tailwind CSS.
- Sin base de datos, ORM, auth ni capa API por defecto.
- Una configuración de proyecto generada que puedes extender más adelante.

## Forma generada

Esta guía empieza intencionalmente con una app Qwik City solo frontend. Es útil cuando quieres tener lista la carcasa de aplicación, routing y estilos antes de elegir backend o persistencia.

Forma representativa:

```txt
my-qwik-app/
  bts.jsonc
  src/
    routes/
    components/
```

El `bts.jsonc` generado registra que backend, runtime, base de datos, ORM, auth y API son todos `none`. Eso hace explícita la naturaleza mínima del stack.

## Ruta de ejemplo

Las rutas Qwik City pueden empezar pequeñas:

```tsx
import { component$ } from "@builder.io/qwik";

export default component$(() => {
  return (
    <main class="mx-auto max-w-4xl p-6">
      <h1 class="text-3xl font-semibold">Qwik app</h1>
      <p class="text-slate-600">Add server features when the product needs them.</p>
    </main>
  );
});
```

Mantén las primeras rutas centradas en UI y requisitos de datos. Agrega backend solo cuando haya una responsabilidad real de servidor, como auth, persistencia, webhooks o llamadas privadas a APIs.

## Compatibilidad

Qwik tiene soporte backend/API más estrecho en Better Fullstack que las opciones React, Nuxt, Svelte o Solid. Esta guía mantiene el stack intencionalmente mínimo y compatible.

Cómo el comando usa `--backend none`, `--database none`, `--orm none`, `--auth none` y `--api none`, no debería generar contratos de servidor ni modelos de base de datos. Si necesitas eso desde el primer día, elige una combinación framework/backend con soporte Better Fullstack más completo.

## Cuándo elegirlo

Elige Qwik para apps frontend-first donde resumability y un payload inicial mínimo importen más que un backend generado desde el día uno.

## Agregar backend después

Cuándo la app necesite datos, decide si Qwik debe seguir como frontend mientras otro servicio es dueño de la API, o si otro preset de Better Fullstack encaja mejor con el requisito fullstack. Para muchos equipos, agregar una API Hono como servicio separado es más limpio que forzar responsabilidades backend dentro de un scaffold frontend-first.

Adiciones futuras útiles:

- Una API Hono para lógica privada de servidor.
- PostgreSQL más Drizzle o Prisma para datos relacionales.
- Better Auth cuando la gestión de cuentas entre en el producto.
- oRPC u otra capa de contratos tipados cuando frontend y API necesiten tipos compartidos.

## Despliegue

Una app Qwik mínima es sencilla de desplegar como app web o frontend capaz de generar estático, según el adapter y hosting generados. Cómo este comando no agrega base de datos ni auth, la configuración de producción se centra sobre todo en build output, rutas y variables públicas.

## Solucion de problemas

- Si esperabas archivos de base de datos, revisa el comando: esta guía pasa intencionalmente `--database none` y `--orm none`.
- Si faltan rutas de auth, el stack usa `--auth none`.
- Si las clases Tailwind no se aplican, inspecciona los content paths de Tailwind y la integración de framework generados.
- Si luego agregas una API, mantén variables públicas de cliente separadas de secretos privados de servidor.
