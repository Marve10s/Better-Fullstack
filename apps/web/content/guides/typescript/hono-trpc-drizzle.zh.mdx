当你想要一个快速的 TypeScript API server，并搭配独立 React frontend 做 typed client-server calls 时，使用这个 stack。

```npm
npm create better-fullstack@latest my-hono-app -- \
  --ecosystem typescript \
  --frontend tanstack-router \
  --backend hono \
  --runtime bun \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## 创建内容

- 一个 TanStack Router frontend。
- 运行在 Bun 上的 Hono backend。
- PostgreSQL 和 Drizzle。
- Better Auth。
- web app 与 API 之间的 tRPC。
- 用于 frontend 和 server packages 的 monorepo structure。

## 生成结构

这个命令创建 split app shape：frontend 和 backend 是分开的 packages，但一起生成并共享同一个 stack configuration。这样 API boundary 很明确，同时保留 TypeScript inference。

代表性结构：

```txt
my-hono-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

具体 workspace 名称可能不同，但架构重点相同：TanStack Router 负责 browser app，Hono 负责 HTTP，Drizzle 负责 database access，tRPC 是它们之间的 typed contract。

## 示例 Hono route

当 request handling 需要保持 explicit 和 lightweight 时，Hono 很合适：

```ts
import { Hono } from "hono";

const app = new Hono();

app.get("/health", (c) => {
  return c.json({ ok: true });
});

export default app;
```

Framework-level middleware、auth context 和 request logging 应留在 server package 中。避免把 Hono request objects 泄漏进 shared business logic。

## 示例 tRPC procedure shape

tRPC procedures 应该是围绕 server operations 的小型 validated entry points：

```ts
import { z } from "zod";

const createTaskInput = z.object({
  title: z.string().min(1),
});

export const taskProcedures = {
  create: async (rawInput: unknown, userId: string) => {
    const input = createTaskInput.parse(rawInput);

    return {
      id: crypto.randomUUID(),
      title: input.title,
      ownerId: userId,
    };
  },
};
```

大型 apps 中，只有当 client 和 server 都需要 validation schemas 时，才把它们放入 shared module。Database writes 应保持 server-only。

## 何时选择

当 backend 需要保持小、快、明确时，选择 Hono。这个配置适合 API-first products、dashboards、internal tools，以及之后可能拆分 frontend/backend deployment 的服务。

## 兼容性

- `--frontend tanstack-router` 加 `--backend hono` 会得到独立 React web app 和 API server。
- 本指南中 `--runtime bun` 为 Hono service 选择 Bun。
- 这里适合 `--api trpc`，因为 frontend 是 React-based。SvelteKit、Nuxt 或 SolidStart guides 中 Better Fullstack 使用 oRPC。
- PostgreSQL 和 Drizzle 是适合这种 split service shape 的 production-oriented 组合。

## 部署

如果 hosting 把 frontend 和 Hono server 当成两个 targets，按两个 deployable targets 规划。认真配置 CORS、auth cookie domains 和 public API URLs；split services 会把这些细节显式暴露出来。

API server 需要 PostgreSQL connection string 和 auth secrets。web app 需要 API origin 和任何 public app URL values。不要把 secrets 放进 frontend build-time public variables。

## 故障排查

- 如果 browser requests 在到达 Hono 前失败，检查 CORS 和配置的 API base URL。
- 如果跨域时 auth cookies 缺失，检查 same-site settings、secure cookies，以及 web/API domains 是否符合预期部署。
- 如果 tRPC calls 可以编译但 runtime 失败，确认 Hono tRPC adapter 挂载在 client 使用的同一路径。
- 如果生成的 apps 无法连接本地 PostgreSQL，确认 database 正在运行，并且 connection string 匹配所选 database。

## Tradeoffs

如果你想让一个 framework 管完整个 app，使用 [TanStack Start](/guides/typescript/create-tanstack-start-project/) 或 [Next.js](/guides/typescript/nextjs-drizzle-better-auth/)。如果你想要独立 API boundary，Hono 是更清晰的默认选择。

相比 self-backend fullstack framework，这个 stack 的 moving parts 更多。回报是更干净的 service boundary、更容易做 API-focused testing，以及之后可以服务多个 clients 的 backend。
