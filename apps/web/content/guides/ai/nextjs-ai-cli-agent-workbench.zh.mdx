当你希望 Next.js 项目一开始就支持 AI 辅助开发、本地 MCP 工具和自动生成的代理说明文档时，使用这个技术栈。

```npm
npm create better-fullstack@latest my-agent-workbench -- \
  --ecosystem typescript \
  --frontend next \
  --backend self \
  --database sqlite \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --ai ai-cli \
  --addons mcp skills \
  --ai-docs claude-md agents-md cursorrules \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## 创建内容

- 一个带本地全栈配置的 Next.js 应用。
- SQLite 和 Drizzle。
- Better Auth。
- AI CLI 配置。
- MCP 和 Skills add-ons。
- 面向代理的说明文档，例如 `CLAUDE.md`、`AGENTS.md` 或 `.cursorrules`，具体取决于所选 flags。

## 生成结构

这个预设既会生成应用本身，也会生成代理工作流所需的配套文件。Web 应用仍是标准的 Next.js 全栈项目，附加项则为代码代理和本地 AI 工具补充仓库上下文。

代表性结构：

```txt
my-agent-workbench/
  bts.jsonc
  AGENTS.md
  CLAUDE.md
  .cursorrules
  app/
  components/
  lib/
    auth/
    db/
```

具体说明文档取决于 `--ai-docs claude-md agents-md cursorrules`。示例命令请求全部三种格式，这样不同代理工具都能在自己偏好的格式中找到项目指导。

## AI 相关部分如何配合

`--ai ai-cli` 会启用 AI CLI 支持。`--addons mcp skills` 会加入围绕 MCP 和 Skills 的本地代理能力。`--ai-docs` 则控制生成哪些仓库级说明文件。

这些部分各自承担不同职责：

| 部分 | 作用 | 典型用途 |
| --- | --- | --- |
| AI CLI | 本地命令行 AI 流程 | 运行由代理辅助的项目任务 |
| MCP | 工具和上下文桥接 | 向代理暴露项目或服务能力 |
| Skills | 可复用的代理指令 | 标准化调试或功能开发等流程 |
| Agent docs | 仓库级说明 | 告诉代理项目如何组织 |

## 示例代理指令

面向代理的说明文档应该具体。优先写可执行、可验证的项目规则：

```md
## Verification

After editing authentication, run the smallest auth-related test or typecheck.
Do not run broad workspace commands unless the change crosses package boundaries.

## Project boundaries

Database access belongs in server-only modules.
Client components should call typed API helpers instead of importing Drizzle code.
```

当架构变化时，保持这些说明文档最新。过期的代理指令比没有指令更糟，因为它们会自信地把后续工作引向错误文件。

## 示例应用模型

应用侧仍然是普通的 Better Fullstack TypeScript 技术栈。一个小型 SQLite 表可以这样写：

```ts
import { integer, sqliteTable, text } from "drizzle-orm/sqlite-core";

export const agentRun = sqliteTable("agent_run", {
  id: text("id").primaryKey(),
  goal: text("goal").notNull(),
  status: text("status").notNull(),
  createdAt: integer("created_at", { mode: "timestamp" }).notNull(),
});
```

应用表应该用于产品数据，不要用来存放 secrets。MCP credentials、provider tokens 和 AI service keys 应放在环境变量或供应商专用的安全存储中。

## 何时选择

当仓库本身需要容易被代码代理理解和修改时，选择这个技术栈。它适合需要标准化代理流程、MCP 工具和生成式项目上下文的团队。

## 兼容性

- 这个预设使用 TypeScript Next.js 应用和 `--backend self`，因此 Next.js 负责服务端边界。
- `--addons mcp skills` 是开发工作流附加项。它们不会替代应用的认证、数据库或 API 决策。
- `--ai-docs claude-md agents-md cursorrules` 会有意为不同工具生成多种指令格式。
- SQLite 对本地 workbench 类项目很方便。如果 workbench 变成带共享状态的多用户生产产品，请使用 PostgreSQL。

## 部署

像普通全栈应用一样部署 Next.js app：配置 Better Auth secrets、app URL 和 database settings。把 AI provider keys、MCP credentials 和任何 tool tokens 都当作服务端 secrets。

代理说明文档通常应该进入版本控制。本地凭据、生成日志和机器专属 MCP 配置通常不应该进入。提交前检查生成的 ignore 文件。

## 故障排查

- 如果 AI 工具忽略仓库说明，确认它会读取你用 `--ai-docs` 选择生成的文件名。
- 如果 MCP 工具不可用，检查本地 MCP 设置，以及使用它的代理是否指向该设置。
- 如果生成的 addon setup 在脚本化运行中弹出 prompts，像本指南一样使用显式 flags，让 defaults 可预测。
- 如果 app 本地可用但部署后 auth 失败，检查任何 Next.js 技术栈中都要检查的 Better Auth URL 和 cookie settings。

## 取舍

这个技术栈比最小 Next.js starter 更有主见。如果你只需要产品应用，不需要代理工具，使用 [Next.js with Drizzle and Better Auth](/guides/typescript/nextjs-drizzle-better-auth/)。

当人类和代理会反复在仓库中协作时，额外文件很有价值。对于一次性原型，它们是额外负担。如果团队不会维护这些指令，先从更简单的 Next.js guide 开始，之后再添加 AI docs。
