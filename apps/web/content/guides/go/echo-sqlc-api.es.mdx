Usa este stack cuando quieras una API Go con Echo y acceso a datos SQL-first.

```npm
npm create better-fullstack@latest my-echo-api -- \
  --ecosystem go \
  --go-web-framework echo \
  --go-orm sqlc \
  --go-api none \
  --go-cli none \
  --go-logging zap \
  --go-auth none
```

## Qué crea

- Un proyecto API Go usando Echo.
- SQLC como opción de acceso a base de datos.
- Zap para logging.
- Sin gRPC, CLI ni add-on de auth por defecto.

## Forma generada

Este stack es una API Go SQL-first. Echo proporciona la capa HTTP, SQLC genera métodos y tipos Go desde SQL, y Zap maneja logs estructurados.

El objetivo principal de diseño es mantener visible el comportamiento de base de datos:

- Los archivos SQL definen las consultas.
- SQLC genera tipos Go y métodos de query.
- Los handlers llaman a servicios o stores en vez de incrustar SQL.
- El middleware de Echo maneja preocupaciones de request como logging, recovery y CORS.

## Árbol representativo

```txt
my-echo-api/
  go.mod
  sqlc.yaml
  queries/
    users.sql
  migrations/
    001_create_users.sql
  cmd/
    server/
      main.go
  internal/
    db/
      generated/
    handlers/
      users.go
    services/
      users.go
```

Mantén SQL bajo control de versiones y revisa los cambios de consultas con el mismo cuidado que el código Go. SQLC hace más seguro el lado Go, pero no reemplaza el diseño de schema.

## Ejemplos de query y handler

Un archivo de queries SQLC es la fuente de verdad para los métodos generados:

```sql
-- name: GetUser :one
select id, email, name, created_at
from users
where id = $1;

-- name: CreateUser :one
insert into users (email, name)
values ($1, $2)
returning id, email, name, created_at;
```

Un handler de Echo puede bindear JSON y llamar a la capa de queries generada a través de un servicio:

```go
package handlers

import (
	"net/http"
	"strconv"

	"github.com/labstack/echo/v4"
)

func (h *UserHandler) Get(c echo.Context) error {
	id, err := strconv.ParseInt(c.Param("id"), 10, 64)
	if err != nil {
		return echo.NewHTTPError(http.StatusBadRequest, "invalid user id")
	}

	user, err := h.Users.Get(c.Request().Context(), id)
	if err != nil {
		return echo.NewHTTPError(http.StatusNotFound, "user not found")
	}

	return c.JSON(http.StatusOK, user)
}
```

Registra las rutas cerca del arranque para que la superficie de la API sea fácil de revisar:

```go
func RegisterRoutes(e *echo.Echo, users *UserHandler) {
	e.GET("/health", Health)

	api := e.Group("/api")
	api.GET("/users/:id", users.Get)
	api.POST("/users", users.Create)
}
```

## Notas de compatibilidad

Esta guía usa intencionalmente `--go-api none`, `--go-cli none` y `--go-auth none` para mantener el scaffold enfocado en una API HTTP. Mantén `--go-orm sqlc` cuando quieras código de consultas generado desde SQL. Usa la guía gRPC si la frontera de servicio debe ser protobuf en vez de JSON.

## Cuándo elegirlo

Elige Echo con SQLC cuando quieras SQL explícito y tipos Go generados sin adoptar un ORM de más alto nivel.

## Tradeoffs

SQLC mantiene visible el comportamiento de base de datos, pero escribes más SQL. Si quieres un flujo ORM convencional, usa [Gin con PostgreSQL y GORM](/guides/go/gin-postgres-gorm/).

Echo es ligero y flexible, así que las convenciones del proyecto importan. Sé consistente sobre dónde viven la validación de requests, las llamadas a servicios y el mapeo de errores.

## Pruebas y despliegue

Ejecuta pruebas Go para handlers y servicios, e incluye pruebas de integración con una base real para las consultas importantes.

```bash
go test ./...
```

Cuándo cambie SQL, regenera la salida SQLC con el comando documentado en el proyecto generado antes de confiar en el feedback del compilador. En despliegue, ejecuta migraciones antes de iniciar la API para que los tipos de query generados coincidan con el schema vivo.

## Troubleshooting

- Si el código SQLC generado está obsoleto, vuelve a ejecutar el comando de generación SQLC del proyecto y commitea tanto SQL como cambios Go generados si el scaffold versiona esos archivos.
- Si una query compila pero devuelve datos inesperados, revisa primero el SQL; SQLC valida tipos, no semántica de producto.
- Si Echo devuelve errores en texto plano, centraliza el manejo de errores HTTP para que las respuestas sean consistentes.
- Si `go test ./...` falla solo en pruebas de base de datos, verifica la URL de la base de test y el estado de las migraciones.
