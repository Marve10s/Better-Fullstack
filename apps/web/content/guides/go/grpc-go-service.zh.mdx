当你的 Go 服务需要 gRPC 契约，并且希望从一开始就获得生成好的后端项目结构时，使用这个 stack。

```npm
npm create better-fullstack@latest my-go-grpc -- \
  --ecosystem go \
  --go-web-framework gin \
  --go-orm gorm \
  --go-api grpc-go \
  --go-cli none \
  --go-logging zap \
  --go-auth none
```

## 创建内容

- 使用 Gin 的 Go 后端。
- gRPC-Go 作为 API 选项。
- GORM 用于数据库访问。
- Zap 用于日志记录。

## 生成结构

这个 stack 会创建一个带类型化 gRPC 边界的 Go 服务。Gin 仍然可以提供 health check 或其他运维 HTTP endpoint，而 gRPC-Go 负责其他后端客户端依赖的服务契约。

服务最容易维护的方式是：

- Protobuf 文件定义 request 和 response 契约。
- gRPC handler 将传输层输入转换为服务调用。
- GORM model 留在持久化层。
- Gin route 与 gRPC 服务表面保持分离。

## 代表性文件树

```txt
my-go-grpc/
  go.mod
  proto/
    users/v1/users.proto
  cmd/
    server/
      main.go
  internal/
    grpc/
      users.go
    http/
      health.go
    models/
      user.go
    services/
      users.go
```

把生成的 protobuf 输出放在 scaffold 预期的位置，并避免直接编辑生成文件。

## Proto 与服务示例

protobuf service 应该描述客户端真正需要的契约：

```proto
syntax = "proto3";

package users.v1;

option go_package = "my-go-grpc/internal/gen/users/v1;usersv1";

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}

message GetUserRequest {
  int64 id = 1;
}

message User {
  int64 id = 1;
  string email = 2;
  string name = 3;
}
```

Go 实现应当把领域错误映射成 gRPC 状态码：

```go
package grpc

import (
	"context"

	usersv1 "my-go-grpc/internal/gen/users/v1"

	"google.golang.org/grpc/codes"
	"google.golang.org/grpc/status"
)

type UserServer struct {
	usersv1.UnimplementedUserServiceServer
	Users *UserService
}

func (s *UserServer) GetUser(
	ctx context.Context,
	req *usersv1.GetUserRequest,
) (*usersv1.User, error) {
	user, err := s.Users.Get(ctx, req.GetId())
	if err != nil {
		return nil, status.Error(codes.NotFound, "user not found")
	}

	return &usersv1.User{
		Id:    user.ID,
		Email: user.Email,
		Name:  user.Name,
	}, nil
}
```

需要时可以用 Gin 提供运维 endpoint：

```go
func RegisterHTTP(router *gin.Engine) {
	router.GET("/health", func(c *gin.Context) {
		c.JSON(200, gin.H{"status": "ok"})
	})
}
```

## 兼容性说明

当你需要生成好的 gRPC wiring 时，保留 `--go-api grpc-go`。如果改成 `--go-api none`，项目会变成普通 HTTP 后端，protobuf flow 不再属于 scaffold。`--go-cli none` 让项目保持为服务，而不是命令行工具。

## 何时选择

当后端服务更看重强类型服务契约，而不是面向浏览器的 JSON route 时，选择这个 stack。

## 取舍

gRPC 会增加契约维护和客户端生成流程。如果只需要更简单的 HTTP API，可以从 [Gin with PostgreSQL and GORM](/guides/go/gin-postgres-gorm/) 开始。

GORM 让持久化层更容易上手，但 protobuf 契约不应该机械复制数据库模型。把 `.proto` 文件当作公开服务契约，把 GORM model 当作内部存储细节。

## 测试与部署

先直接测试 service method，再用 test server 为重要的客户端可见行为添加 gRPC 集成测试。

```bash
go test ./...
```

部署时暴露客户端需要的 gRPC port，并将 HTTP health check 保持在平台探测的端口。如果 protobuf generation 是 build 的一部分，确保 CI 安装所需的 generator tool。

## 故障排查

- 如果客户端收到 `unknown service`，确认 gRPC server 注册了生成的服务实现。
- 如果生成的 protobuf package 无法正常 import，检查 `go_package` 选项和 scaffold 的输出路径。
- 如果 health check 正常但 gRPC 失败，确认容器或平台暴露的是 gRPC listener，而不只是 Gin listener。
- 如果数据库错误泄漏给客户端，请明确映射为 `codes.NotFound`、`codes.InvalidArgument` 或 `codes.Internal`。
