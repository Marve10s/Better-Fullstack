当你需要一个 Go HTTP API，并且希望它有熟悉的路由器、关系型数据库访问，以及务实清晰的项目结构时，使用这个 stack。

```npm
npm create better-fullstack@latest my-go-api -- \
  --ecosystem go \
  --go-web-framework gin \
  --database postgres \
  --go-orm gorm \
  --go-logging zap
```

## 创建内容

- 一个使用 Gin 的 Go API 项目。
- PostgreSQL 作为数据库选项。
- GORM 用于数据库访问。
- Zap 用于结构化 logging。
- 一个可以直接开始 backend development 的生成项目布局。

## 生成结构

这个 stack 会创建一个常规的 Go HTTP API。Gin 处理路由和 middleware，GORM 负责模型映射与持久化，PostgreSQL 是后端数据存储，Zap 提供结构化日志。

生成应用里最清晰的边界是：

- HTTP handlers 解析 input 并返回 response。
- Services 承担业务决策。
- GORM models 描述持久化记录。
- Database setup 和 logger setup 只在 startup 时执行一次。

生成的 Go 项目会随 option set 变化，但应用增长时应保持这种形状：framework code 靠近 handlers，persistence code 放在 services 或 repositories 后面。

## Handler 和 model 示例

Gin handler 应该绑定 request data，调用应用代码，并返回清晰的 HTTP status：

```go
package handlers

import (
	"net/http"

	"github.com/gin-gonic/gin"
)

type CreateUserRequest struct {
	Email string `json:"email" binding:"required,email"`
	Name  string `json:"name" binding:"required"`
}

func (h *UserHandler) Create(c *gin.Context) {
	var input CreateUserRequest
	if err := c.ShouldBindJSON(&input); err != nil {
		c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
		return
	}

	user, err := h.Users.Create(c.Request.Context(), input.Email, input.Name)
	if err != nil {
		c.JSON(http.StatusInternalServerError, gin.H{"error": "create user"})
		return
	}

	c.JSON(http.StatusCreated, user)
}
```

对应的 GORM model 可以保持小而明确：

```go
package models

import "time"

type User struct {
	ID        uint      `json:"id" gorm:"primaryKey"`
	Email     string    `json:"email" gorm:"uniqueIndex;not null"`
	Name      string    `json:"name" gorm:"not null"`
	CreatedAt time.Time `json:"createdAt"`
	UpdatedAt time.Time `json:"updatedAt"`
}
```

注册路由时，应把相关 endpoints 放在一起：

```go
func RegisterRoutes(router *gin.Engine, users *UserHandler) {
	router.GET("/health", Health)

	api := router.Group("/api")
	api.POST("/users", users.Create)
	api.GET("/users/:id", users.Get)
}
```

## 兼容性

<GuideCompatibilityNote kind="supported" title="兼容性规则">
  本指南流程需要保持 `--go-web-framework gin`、`--database postgres` 和 `--go-orm gorm` 组合。如果改成 `--go-orm sqlc`，应期待 SQL files 和 generated query types，而不是 GORM models。如果设置 `--go-web-framework none`，项目会变成非 server project，Gin handler 示例也不再适用。
</GuideCompatibilityNote>

## 何时选择它

当你想要一个直接的 Go API，并且需要常见 middleware 模式与较低的 framework overhead 时，选择 Gin。

## 取舍

Gin 和 GORM 对很多 Go 团队来说都是熟悉的选择。如果你偏好更小的 router surface 或不同的 API 风格，可以在 builder 中比较其他 Go framework options。

GORM 对 CRUD-heavy 应用很高效，但它也可能隐藏 SQL 细节。对于查询密集、且精确 SQL 是接口契约一部分的服务，可以对比 Echo with SQLC 指南。

## 测试与部署

用 `httptest` 测试 handlers，并用真实 PostgreSQL database 做围绕 GORM behavior 的 integration tests。

```bash
go test ./...
```

部署前确认 generated app 从环境读取 database URL，并确认 migrations 或 auto-migration behavior 符合团队 release policy。使用 Zap fields 记录 request IDs、user IDs 和 dependency failures，这样生产日志可以不用解析字符串就完成过滤。

## 故障排查

- 如果 requests 返回 400，先检查 Gin binding tags，再去调试 service code。
- 如果 insert 因 duplicate key errors 失败，请同时检查 GORM indexes 和 PostgreSQL constraints。
- 如果 app 无法连接 PostgreSQL，确认 host、port、SSL mode，以及生成的环境变量名称。
- 如果 logs 不是结构化的，确保 Zap logger 被传入 handlers 和 services，而不是使用 `fmt.Println`。
