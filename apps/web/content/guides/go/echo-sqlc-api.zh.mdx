当你需要一个使用 Echo、并采用 SQL-first 数据访问方式的 Go API 时，使用这个 stack。

```npm
npm create better-fullstack@latest my-echo-api -- \
  --ecosystem go \
  --go-web-framework echo \
  --go-orm sqlc \
  --go-api none \
  --go-cli none \
  --go-logging zap \
  --go-auth none
```

## 创建内容

- 使用 Echo 的 Go API 项目。
- SQLC 作为 Go 的 ORM/database access 选项。
- Zap 用于日志记录。
- 默认不包含 gRPC、CLI 或 Go auth add-on。

## 生成结构

这是一个 SQL-first Go API。Echo 提供 HTTP layer，SQLC 从 SQL 生成 typed Go method，Zap 负责结构化日志。

主要设计目标是让数据库行为保持可见：

- SQL 文件定义 query。
- SQLC 生成 Go 类型和 query method。
- Handler 调用 service 或 store，而不是直接嵌入 SQL。
- Echo middleware 处理 logging、recovery、CORS 等 request 关注点。

## 代表性文件树

```txt
my-echo-api/

  queries/

  cmd/

      main.go
  internal/

    handlers/
      users.go
    services/
      users.go
```

把 SQL 放入 version control，并像 review Go code 一样认真 review query change。SQLC 让 Go 侧更安全，但不能代替 schema design。

## Query 与 handler 示例

SQLC query 文件是生成方法的 source of truth：

```sql
-- name: GetUser :one
select id, email, name, created_at
from users
where id = $1;

-- name: CreateUser :one
insert into users (email, name)
values ($1, $2)
returning id, email, name, created_at;
```

Echo handler 可以 bind JSON，并通过 service 调用生成的 query layer：

```go
package handlers

import (
	"net/http"
	"strconv"

	"github.com/labstack/echo/v4"
)

func (h *UserHandler) Get(c echo.Context) error {
	id, err := strconv.ParseInt(c.Param("id"), 10, 64)
	if err != nil {
		return echo.NewHTTPError(http.StatusBadRequest, "invalid user id")
	}

	user, err := h.Users.Get(c.Request().Context(), id)
	if err != nil {
		return echo.NewHTTPError(http.StatusNotFound, "user not found")
	}

	return c.JSON(http.StatusOK, user)
}
```

把 route 注册放在靠近启动代码的位置，这样 API surface 更容易浏览：

```go
func RegisterRoutes(e *echo.Echo, users *UserHandler) {
	e.GET("/health", Health)

	api := e.Group("/api")
	api.GET("/users/:id", users.Get)
	api.POST("/users", users.Create)
}
```

## 兼容性说明

本指南故意使用 `--go-api none`、`--go-cli none` 和 `--go-auth none`，让 scaffold 专注于 HTTP API。当你想从 SQL 生成 query code 时，请保留 `--go-orm sqlc`。如果 service boundary 应是 protobuf 而不是 JSON，请使用 gRPC guide。

## 何时选择

当你想要显式 SQL 和生成的 Go types，而不采用更高层 ORM 时，选择 Echo with SQLC。

## 取舍

SQLC 让 database behavior 可见，但你需要维护更多 SQL。如果想要更传统的 ORM flow，请使用 [Gin with PostgreSQL and GORM](/guides/go/gin-postgres-gorm/)。

Echo 很轻量也很灵活，因此项目约定很重要。请一致地安排 request validation、service call 和 error mapping 的位置。

## 测试与部署

为 handler 和 service 运行 Go test，并为重要 SQL query 加上 database-backed integration test。

```bash
go test ./...
```

SQL 改变时，在信任 compiler feedback 前，先用生成项目记录的命令重新生成 SQLC output。部署时，在启动 API 前运行 migration，确保 generated query type 与 live schema 匹配。

## 故障排查

- 如果生成的 SQLC 代码过期，重新运行项目的 SQLC generation command；如果 scaffold 跟踪生成文件，请同时提交 SQL 和生成的 Go 变更。
- 如果 query 能编译但返回了意外数据，先检查 SQL；SQLC 验证类型，不验证产品语义。
- 如果 Echo 返回纯文本错误，集中处理 HTTP error，让响应保持一致。
- 如果 `go test ./...` 只在数据库测试中失败，检查 test database URL 和 migration 状态。
