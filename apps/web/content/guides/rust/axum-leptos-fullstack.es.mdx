Usa este stack cuando quieras Rust tanto en el servidor como en el frontend de la app generada.

```npm
npm create better-fullstack@latest my-rust-fullstack -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --rust-frontend leptos \
  --rust-orm sea-orm \
  --rust-api none \
  --rust-cli none \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## Qué crea

- Un servidor Rust basado en Axum.
- Leptos como opción frontend Rust.
- SeaORM para acceso a datos.
- Defaults de tracing y manejo de errores estructurado.

## Forma generada

Este stack es una app fullstack Rust-first. Axum controla el runtime del servidor, Leptos controla la superficie frontend, y SeaORM da al backend una capa de persistencia tipada.

La app generada es útil cuando quieres que un solo workspace Rust contenga:

- Startup del servidor, routing y estado compartido.
- Vistas Leptos y código de UI interactiva.
- Entidades de base de datos y acceso a datos server-side.
- Logging y manejo de errores configurados desde el principio.

## Árbol representativo

Espera un layout que separa preocupaciones de servidor y UI mientras las mantiene en un solo proyecto:

```txt
my-rust-fullstack/
  Cargo.toml
  .env.example
  src/
    main.rs
    app.rs
    error.rs
    routes/
      mod.rs
      api.rs
    entities/
      mod.rs
      user.rs
    pages/
      mod.rs
      home.rs
      users.rs
    components/
      mod.rs
```

Usa el árbol generado como guía de límites: los componentes Leptos deberían enfocarse en renderizado e interacción, mientras que el trabajo de base de datos debería permanecer server-side.

## Ejemplos de servidor y UI

Una ruta Axum puede exponer endpoints JSON para acciones server-side o clientes de integración:

```rust
use axum::{extract::State, Json};
use serde::Serialize;

use crate::{error::AppError, AppState};

#[derive(Debug, Serialize)]
pub struct HealthResponse {
    pub status: &'static str,
}

pub async fn health(State(_state): State<AppState>) -> Result<Json<HealthResponse>, AppError> {
    Ok(Json(HealthResponse { status: "ok" }))
}
```

Una página Leptos debería llamar server functions o helpers de API en vez de abrir conexiones de base de datos desde código UI:

```rust
use leptos::prelude::*;

#[component]
pub fn HomePage() -> impl IntoView {
    view! {
        <main>
            <h1>"Dashboard"</h1>
            <p>"Rust server, Rust UI, and one generated project shape."</p>
        </main>
    }
}
```

Para UI respaldada por base de datos, mantén la operación detrás de la frontera del servidor:

```rust
use sea_orm::{DatabaseConnection, EntityTrait};

use crate::entities::user;

pub async fn list_users(db: &DatabaseConnection) -> Result<Vec<user::Model>, sea_orm::DbErr> {
    user::Entity::find().all(db).await
}
```

## Notas de compatibilidad

Esta guía usa flags específicos de Rust en vez de flags de frontend/backend TypeScript. Mantén las opciones de frontend Rust bajo `--rust-frontend`.

Los flags `--rust-api none` y `--rust-cli none` mantienen este scaffold enfocado en la app web. Agrega gRPC o CLI solo cuando el proyecto generado vaya a servir varias interfaces a propósito. Si quieres un backend solo API, usa `--rust-frontend none` y sigue la guía de API Axum.

## Cuándo elegirlo

Elige este stack cuando el proyecto se beneficie de una arquitectura Rust-first a través del límite de app y tu equipo esté cómodo con el modelo compile-time de Rust.

Encaja muy bien en herramientas internas, superficies de admin y áreas de producto donde compartir tipos y validación Rust entre servidor y UI aporta más valor que adoptar un ecosistema frontend JavaScript.

## Tradeoffs

Fullstack Rust reduce el cambio de lenguaje, pero el flujo frontend es distinto de React, Vue o Svelte. Planifica tiempos de compilación Rust, patrones de routing específicos de Leptos y un equipo cómodo depurando comportamiento de navegador desde UI generada por Rust.

## Pruebas y despliegue

Usa pruebas unitarias Rust para servicios y lógica de modelos, y agrega pruebas de rutas o navegador alrededor de los flujos más importantes.

```bash
cargo test
```

En despliegue, verifica ambas mitades: el servidor debe recibir sus variables de entorno, y los assets frontend deben construirse con los scripts del proyecto generado. Mantén migraciones de base de datos en el camino de despliegue si la app guarda datos de usuario.

## Troubleshooting

- Si la UI compila pero faltan datos del servidor, revisa que la ruta server-side o server function esté registrada en el router Axum.
- Si los tipos de base de datos se desalinean, regenera o actualiza entidades SeaORM con el workflow que documenta el scaffold.
- Si una ruta de navegador funciona localmente pero no después del despliegue, revisa juntas la configuración de routing del servidor generado y la de assets estáticos.
- Si los logs están demasiado silenciosos, define `RUST_LOG=info` antes de iniciar el servidor.
