当你希望生成的应用在 server 和 frontend 两侧都使用 Rust 时，使用这个 stack。

```npm
npm create better-fullstack@latest my-rust-fullstack -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --rust-frontend leptos \
  --rust-orm sea-orm \
  --rust-api none \
  --rust-cli none \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## 创建内容

- 基于 Axum 的 Rust server。
- Leptos 作为 Rust frontend 选项。
- SeaORM 用于数据库访问。
- tracing 和结构化错误处理默认配置。

## 生成结构

这个 stack 是 Rust-first 的 fullstack app。Axum 负责 server runtime，Leptos 负责 frontend surface，SeaORM 为后端提供类型化持久化层。

当你希望一个 Rust workspace 同时包含这些内容时，生成的 app 很有用：

- Server startup、routing 和 shared state。
- Leptos view 与交互式 UI code。
- Database entity 与 server-side data access。
- 从一开始就配置好的 logging 和 error handling。

## 代表性文件树

你会看到一个在单个项目中分离 server concern 和 UI concern 的 layout：

```txt
my-rust-fullstack/
  Cargo.toml
  .env.example
  src/
    main.rs
    app.rs
    error.rs
    routes/
      mod.rs
      api.rs
    entities/
      mod.rs
      user.rs
    pages/
      mod.rs
      home.rs
      users.rs
    components/
      mod.rs
```

把生成的树当作边界指南：Leptos component 应该专注渲染和交互，数据库工作应留在 server-side。

## Server 与 UI 示例

Axum route 可以为 server-side action 或 integration client 暴露 JSON endpoint：

```rust
use axum::{extract::State, Json};
use serde::Serialize;

use crate::{error::AppError, AppState};

#[derive(Debug, Serialize)]
pub struct HealthResponse {
    pub status: &'static str,
}

pub async fn health(State(_state): State<AppState>) -> Result<Json<HealthResponse>, AppError> {
    Ok(Json(HealthResponse { status: "ok" }))
}
```

Leptos page 应该调用 server function 或 API helper，而不是从 UI code 打开数据库连接：

```rust
use leptos::prelude::*;

#[component]
pub fn HomePage() -> impl IntoView {
    view! {
        <main>
            <h1>"Dashboard"</h1>
            <p>"Rust server, Rust UI, and one generated project shape."</p>
        </main>
    }
}
```

对于 database-backed UI，把数据库操作放在 server boundary 后面：

```rust
use sea_orm::{DatabaseConnection, EntityTrait};

use crate::entities::user;

pub async fn list_users(db: &DatabaseConnection) -> Result<Vec<user::Model>, sea_orm::DbErr> {
    user::Entity::find().all(db).await
}
```

## 兼容性说明

本指南使用 Rust-specific flag，而不是 TypeScript frontend/backend flag。Rust frontend 选择应放在 `--rust-frontend` 下。

`--rust-api none` 和 `--rust-cli none` 让这个 scaffold 专注于 web app。只有当生成项目明确要服务多个 interface 时，才添加 gRPC 或 CLI 选项。如果你想要 API-only 后端，请使用 `--rust-frontend none` 并参考 Axum API guide。

## 何时选择

当项目受益于跨 app boundary 的 Rust-first 架构，并且团队适应 Rust compile-time model 时，选择这个 stack。

它很适合 internal tool、admin surface，以及那些共享 Rust 类型和 validation 比采用 JavaScript frontend ecosystem 更有价值的产品区域。

## 取舍

Fullstack Rust 减少语言切换，但 frontend workflow 不同于 React、Vue 或 Svelte。需要为 Rust compile time、Leptos-specific routing pattern，以及从 Rust-generated UI 调试浏览器行为的团队能力做计划。

## 测试与部署

对 service 和 model logic 使用 Rust unit test，并围绕最重要的 flow 添加 route 或 browser-level test。

```bash
cargo test
```

部署时验证两部分：server 必须接收环境变量，frontend asset 必须通过 generated project script 构建。如果应用存储用户数据，请把 database migration 放进部署路径。

## 故障排查

- 如果 UI 能构建但缺少 server data，检查 server route 或 server function 是否已注册到 Axum router。
- 如果 database type 发生 drift，请按照 scaffold 文档中的 workflow 重新生成或更新 SeaORM entity。
- 如果浏览器 route 本地可用但部署后不可用，请同时检查生成 server routing 和 static asset 配置。
- 如果日志太少，在启动 server 前设置 `RUST_LOG=info`。
