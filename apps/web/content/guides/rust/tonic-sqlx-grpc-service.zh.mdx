当你需要面向 service-to-service API 和 typed gRPC contract 的 Rust 后端时，使用这个 stack。

```npm
npm create better-fullstack@latest my-rust-grpc -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --rust-frontend none \
  --rust-orm sqlx \
  --rust-api tonic \
  --rust-cli none \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## 创建内容

- 使用 Axum 的 Rust 服务。
- Tonic 用于 gRPC。
- SQLx 用于数据库访问。
- tracing 和结构化错误处理。

## 生成结构

这个 stack 面向类型化的 service-to-service 调用。Tonic 负责 gRPC 契约和 server implementation，SQLx 负责数据库访问；当生成项目包含 Axum 时，它仍然可以提供 health check 等运维 HTTP route。

核心边界是：

- `proto/` 定义服务契约。
- 生成的 Rust gRPC type 靠近 Tonic service implementation。
- SQLx query 放在 repository 或 service module 中，而不是 transport handler 里。
- Axum route 如果存在，应作为运维 endpoint，而不是主要产品 API。


`build.rs` 通常是连接 protobuf generation 的地方。把生成代码和手写业务逻辑分开。

## Proto 与服务示例

一个小的 protobuf 契约让客户端和服务端拥有同一条 typed boundary：

```proto
syntax = "proto3";

package users.v1;

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}

message GetUserRequest {
  int64 id = 1;
}

message User {
  int64 id = 1;
  string email = 2;
  string name = 3;
}
```

Tonic implementation 应该把 transport error 转换为 gRPC status，并把数据库访问留在单独模块中：

```rust
use tonic::{Request, Response, Status};

use crate::grpc::users_v1::{
    user_service_server::UserService, GetUserRequest, User,
};
use crate::repositories::users::UserRepository;

pub struct UsersGrpc {
    repo: UserRepository,
}

#[tonic::async_trait]
impl UserService for UsersGrpc {
    async fn get_user(
        &self,
        request: Request<GetUserRequest>,
    ) -> Result<Response<User>, Status> {
        let id = request.into_inner().id;
        let user = self
            .repo
            .find_by_id(id)
            .await
            .map_err(|error| Status::internal(error.to_string()))?
            .ok_or_else(|| Status::not_found("user not found"))?;

        Ok(Response::new(User {
            id: user.id,
            email: user.email,
            name: user.name,
        }))
    }
}
```

SQLx 让 query shape 保持显式：

```rust
use sqlx::PgPool;

pub struct UserRow {
    pub id: i64,
    pub email: String,
    pub name: String,
}

pub async fn find_user(pool: &PgPool, id: i64) -> Result<Option<UserRow>, sqlx::Error> {
    sqlx::query_as!(
        UserRow,
        "select id, email, name from users where id = $1",
        id
    )
    .fetch_optional(pool)
    .await
}
```

## 兼容性说明

当你需要生成好的 gRPC service wiring 时，保留 `--rust-api tonic`。如果改成 `--rust-api none`，本指南里的 protobuf 和 Tonic 示例就变成手动添加项。SQLx 很适合服务契约，因为 query output 可以直接映射到 protobuf response type。

## 何时选择

当内部服务、微服务或后端系统更适合 gRPC contract，而不是 JSON route 时，选择这个 stack。

## 取舍

gRPC 非常适合强类型服务边界，但会带来协议和 tooling 预期。对于 browser-facing HTTP API，请从 [Axum with PostgreSQL and SeaORM](/guides/rust/axum-postgres-seaorm/) 开始。

SQLx 让数据库行为保持透明，但你需要有意管理 SQL file、migration 和 query compatibility。

## 测试与部署

用 generated client 或 Tonic request type 测试 protobuf-level behavior；当 query correctness 很重要时，用真实 PostgreSQL database 测试 repository function。

```bash
cargo test
```

在 CI 中，如果项目 build 时生成 Rust type，请确保 protobuf tooling 可用。部署时暴露客户端期望的 gRPC port；如果平台需要独立 probe，请将 HTTP health route 与 gRPC service address 分开。

## 故障排查

- 如果 protobuf type 缺失，检查 `build.rs` 并确认 `.proto` path 已包含。
- 如果 SQLx compile-time check 失败，验证生成项目期望的 database URL 或 offline query metadata。
- 如果客户端收到 `UNIMPLEMENTED`，确认 generated service server 已添加到 Tonic server builder。
- 如果 health check 通过但 gRPC call 失败，检查 process manager 或 container 暴露的是否是 gRPC port。
