Usa este stack cuando necesites un backend Rust orientado a APIs servicio-a-servicio y contratos gRPC tipados.

```npm
npm create better-fullstack@latest my-rust-grpc -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --rust-frontend none \
  --rust-orm sqlx \
  --rust-api tonic \
  --rust-cli none \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## Qué crea

- Un servicio Rust usando Axum.
- Tonic para gRPC.
- SQLx para acceso a base de datos.
- tracing y manejo de errores estructurado.

## Forma generada

Este stack está pensado para llamadas tipadas servicio-a-servicio. Tonic controla el contrato gRPC y la implementación del servidor, SQLx controla el acceso a datos, y Axum todavía puede proporcionar rutas HTTP operacionales como health checks cuando el proyecto generado las incluye.

Los límites principales son:

- `proto/` define el contrato del servicio.
- Los tipos gRPC generados en Rust se mantienen cerca de la implementación Tonic.
- Las queries SQLx viven en módulos de repositorio o servicio, no dentro de handlers de transporte.
- Las rutas Axum, si se usan, son endpoints operacionales en vez de la API principal de producto.

## Árbol representativo

```txt
my-rust-grpc/
  Cargo.toml
  build.rs
  proto/

  src/

    error.rs
    grpc/
      mod.rs
      users.rs
    repositories/
      mod.rs
      users.rs
    routes/
      mod.rs

```

El archivo `build.rs` suele ser donde se conecta la generación protobuf. Mantén el código generado fuera de la lógica de negocio editada a mano.

## Ejemplos de proto y servicio

Un contrato protobuf pequeño da a clientes y servidores la misma frontera tipada:

```proto
syntax = "proto3";

package users.v1;

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}

message GetUserRequest {
  int64 id = 1;
}

message User {
  int64 id = 1;
  string email = 2;
  string name = 3;
}
```

La implementación Tonic debería traducir errores de transporte a estados gRPC y mantener el acceso a base de datos en un módulo separado:

```rust
use tonic::{Request, Response, Status};

use crate::grpc::users_v1::{
    user_service_server::UserService, GetUserRequest, User,
};
use crate::repositories::users::UserRepository;

pub struct UsersGrpc {
    repo: UserRepository,
}

#[tonic::async_trait]
impl UserService for UsersGrpc {
    async fn get_user(
        &self,
        request: Request<GetUserRequest>,
    ) -> Result<Response<User>, Status> {
        let id = request.into_inner().id;
        let user = self
            .repo
            .find_by_id(id)
            .await
            .map_err(|error| Status::internal(error.to_string()))?
            .ok_or_else(|| Status::not_found("user not found"))?;

        Ok(Response::new(User {
            id: user.id,
            email: user.email,
            name: user.name,
        }))
    }
}
```

SQLx mantiene explícita la forma de las queries:

```rust
use sqlx::PgPool;

pub struct UserRow {
    pub id: i64,
    pub email: String,
    pub name: String,
}

pub async fn find_user(pool: &PgPool, id: i64) -> Result<Option<UserRow>, sqlx::Error> {
    sqlx::query_as!(
        UserRow,
        "select id, email, name from users where id = $1",
        id
    )
    .fetch_optional(pool)
    .await
}
```

## Notas de compatibilidad

Mantén `--rust-api tonic` cuando quieras wiring generado para servicios gRPC. Si cambias a `--rust-api none`, los ejemplos protobuf y Tonic de esta guía pasan a ser una adición manual. SQLx encaja bien con contratos de servicio porque la salida de queries puede mapearse directamente a tipos de respuesta protobuf.

## Cuándo elegirlo

Elige este stack para servicios internos, microservicios y sistemas backend donde los contratos gRPC encajan mejor que rutas JSON.

## Tradeoffs

gRPC es excelente para fronteras de servicio fuertemente tipadas, pero agrega expectativas de protocolo y tooling. Para APIs HTTP orientadas al navegador, empieza con [Axum con PostgreSQL y SeaORM](/guides/rust/axum-postgres-seaorm/).

SQLx mantiene transparente el comportamiento de base de datos, pero necesitas gestionar SQL files, migraciones y compatibilidad de queries de forma deliberada.

## Pruebas y despliegue

Prueba comportamiento a nivel protobuf con clientes generados o tipos Tonic, y prueba funciones de repository contra PostgreSQL real cuando la corrección de queries importe.

```bash
cargo test
```

En CI, asegúrate de que el tooling protobuf esté disponible si el proyecto genera tipos Rust durante el build. En despliegue, expón el puerto gRPC esperado por clientes y mantén cualquier health route HTTP separada de la dirección del servicio gRPC si tu plataforma requiere probes distintos.

## Troubleshooting

- Si faltan tipos protobuf, revisa `build.rs` y confirma que la ruta `.proto` esté incluida.
- Si las comprobaciones compile-time de SQLx fallan, verifica la database URL o el metadata offline de queries que espera el proyecto generado.
- Si los clientes reciben `UNIMPLEMENTED`, confirma que el generated service server se haya agregado al builder del servidor Tonic.
- Si los health checks pasan pero las llamadas gRPC fallan, revisa que el puerto gRPC sea el que expone tu process manager o contenedor.
